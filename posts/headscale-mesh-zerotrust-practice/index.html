零信任与网络安全 - 这篇文章属于一个选集。
§ : 本文

公司在做一个安全收紧专项，第一刀砍向最大的口子：Prod 数据库的 0.0.0.0/0 公网入口。改成 VPC CIDR 白名单后，开发立刻反馈：“连不上 DB 了，调试咋办？”

第一版方案是 AWS SSM Port Forwarding 封装成脚本，开发本地装 Session Manager Plugin，配 AWS Profile，启动隧道连本地端口。能跑，但每个新人都要踩一遍坑：AWS Profile 切错、SSM Plugin 没装、IAM Group 没加、私有桶下载失败……每开一个新内网资源（Redis、ArgoCD UI、Grafana）都要重新设计一遍访问方案。

最终选定 Headscale + Tailscale 自建零信任 mesh，把所有"如何安全地让开发者访问内网"的问题统一在一个入口下。本文记录从动机、选型、部署、踩坑到落地的完整过程。

一、为什么走到这一步
#

1.1 收紧前的现状
#

收紧前是这样的：

  • Prod Aurora SG 全协议对 0.0.0.0/0 开放
  • 加了一条来源不明的 139.x.x.x/32 白名单（Linode 节点，大概率是某个开发的个人 VPS）
  • 跨 Region 服务通过公网 hostname 直连 Aurora 公网 IP
  • 各种 Redis / 中间件 / 内部 UI 都在 Ingress 暴露
  • 没有任何统一的"谁能访问什么"管控

这种状态下任何一个 SG 规则被攻破、任何一台开发机被入侵，整个数据库的可达性是无限制的。

1.2 收紧的代价
#

把 0.0.0.0/0 删掉很容易。但收紧的同时要保证：

  • 业务服务不中断：跨 Region 服务还得访问，要切到 VPC Peering 或 PrivateLink
  • 开发调试不被卡死：日常查 prod 数据排障的需求是真实的，不能直接禁掉
  • 运维负担可控：单人运维，不能引入需要持续维护的复杂方案

收紧 SG 是一次性操作，但让"开发者可以访问内网资源"这件事可持续运转才是真正难的部分。

1.3 临时方案：SSM Port Forwarding
#

最快能跑起来的是 AWS SSM Port Forwarding：

# 开发者本地执行
aws ssm start-session \
  --target i-xxxxxxxxxxxxx \
  --document-name AWS-StartPortForwardingSessionToRemoteHost \
  --parameters '{
    "host": ["aurora-prod.example.aws.com"],
    "portNumber": ["3306"],
    "localPortNumber": ["13306"]
  }'

# 然后本地连 127.0.0.1:13306
mysql -h 127.0.0.1 -P 13306 -u <user> -p

封装成 db-tunnel.sh 脚本，配 IAM Group db-access + 最小权限 Policy。能跑，但是反复出问题：

  • 开发本地 AWS Profile 配错（用了别的服务账号）
  • 没装 Session Manager Plugin
  • 私有 S3 桶下载脚本时 403
  • 不同操作系统的安装命令不一样
  • 每次开新资源就要写新脚本

更重要的是：这只解决了 DB。Redis 还在公网、Grafana 还在公网、Nacos 还在公网，每个都要单独想方案，运维负担线性增长。

1.4 真正想要的
#

需要一个统一的"谁能访问什么"层，要满足：

  1. 单人能维护：不能再加一套需要 HA + 证书轮换 + 跨云网络打通的复杂控制面
  2. 跨云：AWS 多 Region + 阿里云 都要覆盖
  3. 不依赖 SSO（暂时没有统一 IdP）
  4. 开发体验最好不要比"本地直连"差太多
  5. 权限变更秒级生效（离职、调岗）

二、选型对比
#

调研过的方案：

方案类型100 人成本跨云学习成本自维护负担
AWS Client VPNAWS 原生 VPN约 ¥12k/月❌ 仅 AWS低低
Cloudflare Zero TrustSaaS ZTNA50 人内免费✅低极低
Teleport 自建统一访问代理0 + 重运维✅中高
StrongDM商业 PAM$70+/人/月✅极低0
Pomerium 自建身份感知代理0 + 中等运维✅中中
Headscale + Tailscale自建 mesh VPN≈0（一台 ECS）✅低低

逐个否掉的理由：

  • AWS Client VPN：只覆盖 AWS，阿里云完全不通；按连接小时计费，30 人长期连接成本不低
  • Cloudflare Zero Trust：跨境到 Cloudflare 边缘延迟波动大（中国侧用户体验差）；DB 流量过 CF 边缘有损
  • Teleport：需要 SSO 前置 + HA 控制面 + 证书轮换 + 跨云网络打通，对单人运维就是新债
  • StrongDM：商业方案最贵，30 人按 $70 算约 $25k/年；且仍要 SSO 前置
  • Pomerium：主要是 web 应用代理，对 DB / TCP 的支持不如专门的 mesh 方案

最后选 Headscale + Tailscale 的核心理由：

  1. 零增量成本：Headscale 是 Go 单二进制（50m CPU / 128MB 内存），跑在已有 ECS 上。Tailscale 客户端官方免费
  2. 全协议覆盖：HTTP / TCP / SSH / DB / K8s 都能走 mesh，不局限在 web 应用
  3. 跨云原生：每个 Region/集群部署一个 Subnet Router Pod 即可，不需要跨云网络打通
  4. 不依赖 SSO：可用 PreAuthKey 启动，未来再接 OIDC
  5. 架构简单：单进程 + SQLite 后端就够，100 节点以内毫无压力
  6. ACL 直接是 JSON：改一下 + reload 秒级生效，离职/调岗瞬间断访问

三、整体架构
#

Mesh 类方案的核心设计是控制面 / 数据面分离：

控制面 (Control Plane)数据面 (Data Plane)
组件Headscale 服务WireGuard 加密通道
职责决定谁能和谁通信实际承载数据流量
流量特征小、低频（HTTPS API）大、高频（UDP/TCP）

Headscale 不传数据，只发"通讯录"。

3.1 控制面：Headscale
#

部署在阿里云 cn-beijing 一台 2c4g ECS 上，Docker Compose 起两个容器：

  • headscale：Go 单二进制，监听 8080（内部）
  • caddy：反向代理 + 自动 Let’s Encrypt 签证，对外 443

数据面有四种状态需要控制面下发：

  1. 节点列表（每个节点的公钥 + 100.64.x.x 内部 IP）
  2. ACL 规则（编译后的 packetfilter）
  3. 路由（哪个 Subnet Router 负责哪个 CIDR）
  4. DERP 服务器列表（NAT 打洞失败时的中继）

控制面挂了的影响：新连接建不起来，已建立的 mesh 连接照常工作（WireGuard 是无状态加密协议）。

3.2 数据面：WireGuard + DERP
#

WireGuard 是 Linux 内核里 4000 行代码的现代加密协议，比 OpenVPN 快 4-10 倍。它走 UDP（默认 41641），加密用 ChaCha20-Poly1305。

但 UDP 在某些网络环境会被 QoS 或拦截（典型如国内某些运营商）。这时 Tailscale 自动 fallback 到 DERP 中继：

  • DERP 是 Tailscale 设计的简单中继协议，跑在 HTTPS (TCP 443) 上
  • DERP 服务器看不到加密内容，只是个"加密管道转发器"
  • Headscale 内嵌一个 DERP，也支持加独立 DERP relay

跨境（中→美）实测延迟约 175ms（光速极限），抖动小，比走公网 IP 直连稳定。

3.3 关键节点：Subnet Router
#

普通 mesh 节点之间互通，但 mesh 节点访问 VPC 内的资源（如 Aurora 内网 IP）需要"网关"。这就是 Subnet Router：

  • 跑在 K8s 集群里的一个普通 Pod，启用 IP forwarding
  • 跑 Tailscale 客户端，advertise 集群所在 VPC 的 CIDR
  • mesh 用户访问 10.x.y.z 时，包先到 Subnet Router 再转发到 VPC 内目标
apiVersion: apps/v1
kind: Deployment
metadata:
  name: subnet-router
spec:
  template:
    spec:
      containers:
      - name: tailscale
        image: tailscale/tailscale:v1.84.0
        env:
        - name: TS_KUBE_SECRET
          value: "tailscale-state"
        - name: TS_USERSPACE
          value: "false"
        - name: TS_HOSTNAME
          value: "subnet-router-prod"
        - name: TS_ROUTES
          value: "10.x.0.0/18"   # 只 advertise VPC CIDR
        - name: TS_EXTRA_ARGS
          value: "--login-server=https://headscale.example.cn --accept-routes"
        - name: TS_AUTHKEY
          valueFrom:
            secretKeyRef:
              name: tailscale-auth
              key: TS_AUTHKEY
        securityContext:
          capabilities:
            add: ["NET_ADMIN"]
        resources:
          requests: {cpu: 50m, memory: 64Mi}
          limits: {cpu: 500m, memory: 256Mi}

3.4 一次完整请求的链路
#

开发本机执行 mysql -h 10.x.y.z -u user -p 时发生的事：

1. mysql 发起 TCP 连接：dst=10.x.y.z:3306
   ↓
2. macOS/Linux 内核查路由表：10.x.0.0/18 → utun (Tailscale TUN 接口)
   ↓
3. 数据包送进 tailscaled 进程
   ↓
4. tailscaled 查"谁负责 10.x.0.0/18？" → Subnet Router (100.64.0.10)
   ↓
5. WireGuard 加密包，发往 Subnet Router
   ↓
6. P2P 直连尝试 → NAT 打不通 → 走 DERP 中继 (TCP 443)
   ↓
7. Subnet Router Pod 解密 WireGuard 包
   ↓
8. Pod 内核启用 IP forwarding，从 tailscale0 转发到 eth0
   ↓
9. 在 VPC 内路由到 Aurora 真实 IP，3306 端口
   ↓
10. 响应原路返回

整条链路全程加密，连中继的 DERP 服务器也看不到包内容。

四、落地实战
#

4.1 阶段划分
#

Phase 0：本地端到端验证（1-2 天）
  ├─ Headscale 控制面部署
  ├─ 一台 K8s 集群（sandbox）部署 Subnet Router
  ├─ 自己客户端接入测试
  └─ ACL / 用户回收 / 跨 Region 验证

Phase 1：生产集群部署（1 周）
  ├─ us-prod 部署 Subnet Router
  ├─ ap-pre 部署 Subnet Router  
  └─ cn-prod 部署 Subnet Router

Phase 2：种子用户试用（2-3 周）
  └─ 拉 2-3 个核心开发，按 ACL 分组试用

Phase 3：全员推广 + 旧方案下线（4-6 周）
  └─ SSM 隧道作为 fallback 保留 1 个月

Phase 4：长期演进
  ├─ 接钉钉 OIDC 做自助登录
  └─ Litestream 备份 + 冷备镜像 + RTO 演练

4.2 控制面部署要点
#

阿里云 ECS 选型：2c4g 按量付费，公网带宽 5Mbps 按使用。月成本约 ¥150。

域名走阿里云 DNS 加 A 记录（控制面在国内访问要快），TLS 证书用 Caddy 自动从 Let’s Encrypt 签发（HTTP-01 挑战需要 SG 开 80 + 443）。

Headscale 配置文件几个关键点：

server_url: https://headscale.example.cn
listen_addr: 0.0.0.0:8080

prefixes:
  v4: 100.64.0.0/10
  v6: fd7a:115c:a1e0::/48
  allocation: sequential

derp:
  server:
    enabled: true   # 内嵌 DERP，单点起步可接受
    region_id: 999
    region_code: "headscale-cn"
  auto_update_enabled: false   # 禁止从 Tailscale 官方拉 DERP 列表

policy:
  mode: file        # ACL 走文件 + git 管理
  path: /etc/headscale/acl.json

ACL 起步配置：

{
  "groups": {
    "group:ops":  ["alice@example.cn"],
    "group:dev":  ["bob@example.cn", "charlie@example.cn"]
  },
  "tagOwners": {
    "tag:subnet-router": ["group:ops"]
  },
  "acls": [
    { "action": "accept", "src": ["group:ops"], "dst": ["*:*"] },
    { "action": "accept", "src": ["group:dev"], "dst": [
        "10.x.y.z/32:3306",     // 仅 prod Aurora
        "10.x.0.0/16:*"         // sandbox 全部
    ]}
  ]
}

4.3 用户与基础设施分离
#

一开始我把 Subnet Router 也注册到我自己的 user 名下，结果 tailscale status 看到 4 个我名下的节点（2 个真实设备 + 2 个 Subnet Router Pod），后期审计很乱。

最佳实践：建一个 infra@example.cn 服务账号专门管基础设施节点，真人 user 只挂个人设备：

alice@example.cn  (运维真人)
├── Mac
└── 远程开发机

bob@example.cn  (开发真人)
└── Mac

infra@example.cn  (服务账号，不参与 ACL src)
├── subnet-router-us-prod
├── subnet-router-ap-pre
└── subnet-router-cn-prod

迁移时注意：新 Subnet Router Pod 要清除旧的 tailscale-state K8s Secret，否则会用旧机器密钥继续注册成旧 user 名下。

# 正确的迁移流程
kubectl create secret generic tailscale-auth -n tailscale-system \
  --from-literal=TS_AUTHKEY="<infra user 的 reusable PreAuthKey>" \
  --dry-run=client -o yaml | kubectl apply -f -

kubectl delete secret tailscale-state -n tailscale-system  # 关键：清旧 state
kubectl rollout restart deploy/subnet-router -n tailscale-system

# 等新节点注册成功，再批准路由
headscale nodes approve-routes --identifier <new-id> --routes 10.x.0.0/18

# 最后再删旧节点
headscale nodes delete --identifier <old-id> --force

五、两个真踩到的坑
#

5.1 CIDR 冲突：mesh 路由抢占 docker bridge
#

部署完 sandbox-staging Subnet Router 后做端到端测试，在控制面 ECS 上跑了一个 MySQL 容器作为目标，从 Mac 连不上。诡异的是同主机 22 端口能通，3306 不通。

排查发现 Subnet Router 同时 advertise 了：

  • 10.x.0.0/16（VPC CIDR，正常）
  • 172.20.0.0/16（K8s ClusterIP CIDR）

而 docker compose 给 test-mysql 项目自动分配的 bridge 网段恰好是 172.20.0.0/16。冲突路径：

Mac 发 SYN → ECS:13306
ECS docker-proxy DNAT 到 172.20.0.2:3306
ECS 内核查路由：172.20.0.0/16 → tailscale0 接口
SYN 被发回 mesh，跑去了 AWS 集群
对端没有 172.20.0.2 这个 ClusterIP → 黑洞

修复：撤销 172.20.0.0/16 的 advertise，只保留 VPC CIDR。

根本教训：mesh 永远只 advertise VPC CIDR，不要广播 K8s ClusterIP / Pod CIDR。

理由：

  1. 多集群的 K8s 默认 ClusterIP 经常重复（172.20.0.0/16 是常见默认值），mesh 里同一段被多个 Subnet Router 广播会路由不确定
  2. ClusterIP 是集群内部地址，跨集群访问应该走 Ingress 或 NodePort
  3. K8s ClusterIP CIDR 段（172.x.x.x）和 docker bridge / 各种 NAT 段经常冲突
  4. 如果真要跨集群访问 ClusterIP，用 Tailscale 4via6 把重复 IPv4 映射到唯一 IPv6（但开发体验差）或重新规划集群 ClusterIP CIDR

5.2 节点删除导致 peer 不同步
#

第一次做"用户回收"测试，直接删掉了 Mac 节点：

headscale nodes delete --identifier 1 --force

然后用新 PreAuthKey 重新接入。Mac 拿到新的 mesh IP 100.64.0.4，tailscale status 看到所有 peer，tailscale ping 能到对端 100.64.0.2，但 TCP 完全不通——nc 全部超时。

从 ECS 端看，“rx 15984 tx 0”——只收不回。OS 层完全没反应，但应用层有连接尝试。

这是 Headscale 的已知 bug (#2693, #1705, #2830)：

节点被删除后，原 peer 的 tailscaled 不会自动刷新 WireGuard 端点表。新节点用新公钥注册了，但其他 peer 还以为旧公钥有效，加密包对端解不开 → 单向通信。

社区临时方案是重启相关 peer 的 tailscaled。但生产环境不能每次开发离职/换电脑都让所有 Subnet Router Pod 重启。

根本修复：生产权限回收用 ACL 而非删节点。

# ❌ 错的（触发 peer 同步 bug）
headscale nodes delete --identifier 1

# ✅ 对的（秒级生效，不触发 bug）
# 编辑 ACL，从 group 里移除该用户的 email
# SIGHUP 重载
docker compose kill -s SIGHUP headscale

ACL 重载是亚秒级的：Headscale 重新编译规则 → 通过长连接推送给所有客户端 → 客户端立刻应用 packetfilter。用户的设备依然在 mesh 里，但所有 dst 权限被拒绝。

进一步的兜底：Subnet Router Deployment 加每日定时 rollout restart，深夜业务低峰期强制刷新一次所有 peer 状态。

六、应用场景
#

6.1 替代 SSM Port Forwarding 连数据库
#

之前：

# 终端 1：起 SSM 隧道
~/db-tunnel.sh prod

# 终端 2：连本地端口
mysql -h 127.0.0.1 -P 13306 -u user -p

现在：

# 直接连 VPC 内私有 IP
mysql -h 10.x.y.z -u user -p

少一个终端、不用 AWS Profile、不用装 SSM Plugin、不用记本地端口。

6.2 远程开发机统一入口
#

很多开发不在本地写代码，而是 SSH 到云端开发机。这种场景下两种处理：

  • 专属开发机：每台机器装 Tailscale，注册成独立节点，权限和本地 Mac 等同
  • 共享跳板机：多个 OS 用户共享一台机器，Tailscale 身份是机器级不是 OS 用户级，要避免共享身份混淆审计。可考虑每个 OS 用户用独立容器 + 独立 Tailscale，或者只把跳板机本身接入 mesh，开发用 SSH ProxyJump 透传

6.3 跨云、跨 Region 访问
#

部署在阿里云的服务要访问 AWS 的资源（典型如 OA 系统跨 Region 访问主力 DB）：

之前要么走公网 hostname，要么单独建跨 Region VPC Peering + Route53 Private Hosted Zone。Mesh 上去之后，所有跨云跨 Region 访问都走同一张网，应用层用 VPC IP 透明可达。

6.4 Claude Code 自动配置接入
#

写了一段标准提示词，开发把它粘进 Claude Code，加上自己的邮箱和 PreAuthKey，cc 自动检测系统、装客户端、登录、验证，全程不到 5 分钟。

[公司内部运维提供的接入流程，已由运维核实，不要再要求 wiki 截图。]

公司零信任 mesh 接入：

【已知事实】
- 控制面：https://headscale.example.cn
- prod DB 内网入口：10.x.y.z:3306
- --accept-routes 是预期配置

【请按顺序执行，每步等我确认】
1. 检测 OS
2. 装 Tailscale
3. sudo tailscale up --login-server=... --auth-key=<key> --accept-routes
4. 验证 tailscale status + nc -zv <db_ip> 3306
5. 持久化 systemctl
6. 写记忆：mesh 节点信息 + 内网资源连接信息

key 一次性消耗，注册成功别重试。报错给方案不要反复确认风险。

提示词开头的"已由运维核实，不要要求 wiki 截图"很关键。第一版没写这句，cc 把整套基础设施当陌生指令反复盘问（域名是不是真的、IP 哪来的、–accept-routes 风险 ack 没有），开发被各种二次确认搞得没耐心。加了授权声明后行为正常多了。

七、最佳实践
#

7.1 CIDR 规划
#

✅ 只 advertise VPC CIDR（10.x.x.x/16-/18）
❌ 不要 advertise K8s ClusterIP CIDR（172.20.x.x 这种段）
❌ 不要 advertise Pod CIDR
❌ 永远不要 advertise 0.0.0.0/0（会抢客户端默认路由）

公司全部 VPC 提前规划保证不重叠（典型 10.0/18, 10.2/18, 10.3/18, 10.50/16 …）
家用网段（192.168.x.x）不要和 VPC 段冲突

7.2 用户与基础设施分离
#

真人 user：只挂个人设备
infra@ 服务账号：只挂 Subnet Router 等基础设施
ACL src 不引用 infra@（基础设施不发起请求，只转发）

7.3 ACL 设计
#

按 tag 写规则比按 CIDR 更灵活：

{
  "tagOwners": {
    "tag:env-prod": ["group:ops"],
    "tag:env-stag": ["group:ops"]
  },
  "acls": [
    {"action": "accept", "src": ["group:dba"],
     "dst": ["tag:env-prod:3306,5432", "tag:env-stag:3306,5432"]},
    {"action": "accept", "src": ["group:dev"],
     "dst": ["tag:env-stag:*"]}
  ]
}

新增 VPC 时给 Subnet Router 节点打对应 tag，无需改 ACL 规则。

7.4 PreAuthKey 发放
#

  • 一次性使用 + 较长有效期（如 90 天），避免开发拿到 key 没立刻用
  • 一对一发放，钉钉/飞书私聊，不群发
  • 服务账号（infra）的 key 用 reusable + 长期，方便 Subnet Router Pod 自愈
  • 写一个 mesh-grant.sh 自动化脚本：建用户 + 改 ACL + 重载 + 生成 key + 输出格式化话术

7.5 用户回收
#

✅ 离职：编辑 ACL 把用户从所有 group 移除 → SIGHUP → 秒级断
✅ 紧急封禁：先 ACL 移除（保住业务），再 headscale nodes expire（不是 delete）
❌ 永远不要 headscale nodes delete 在用节点（触发 peer 同步 bug）

7.6 高可用 / 备份
#

控制面单点是接受的，但要快速恢复：

  • Litestream 实时备份 SQLite 到对象存储
  • Packer 制作冷备 ECS 镜像（含二进制 + 配置）
  • 恢复脚本 + DNS 切换 自动化，目标 RTO 15-30 分钟
  • 每月演练 一次

DERP 容量：30 人最坏情况全员走 DERP，2 vCPU + 1Gbps 起步够用，监控 CPU + 网络 in/out + 并发连接数。

7.7 Subnet Router 加每日重启
#

K8s Deployment annotation 加每日 rollme，避免 peer 状态长期陈旧：

spec:
  template:
    metadata:
      annotations:
        rollme: "{{ daily_timestamp }}"

业务低峰期（深夜）自动 rollout restart，强制刷新所有 peer 关系。Pod 重启期间 mesh 中断 30 秒-1 分钟，可接受。

7.8 推广策略
#

新方案推广不要"一刀切"，遵循 先加能力，再关旧入口：

Week 1-2：sandbox 集群 Subnet Router + 自己试用
Week 3-4：拉 2-3 名种子开发试用
Week 5-6：全员推广 + 文档
Week 7-8：旧 SSM 通道继续保留 1-2 个月作为 fallback
Week 9+：观察无人使用 SSM 后，下线旧通道

八、关键命令速查
#

# === 控制面操作 ===
# 创建用户
headscale users create alice@example.cn

# 生成 PreAuthKey（90 天，一次性）
headscale preauthkeys create --user <id> --expiration 2160h

# 列节点
headscale nodes list

# 批准路由
headscale nodes approve-routes --identifier <id> --routes 10.x.0.0/18

# 让节点过期（不是 delete）
headscale nodes expire --identifier <id>

# ACL 重载
docker compose kill -s SIGHUP headscale

# === 客户端操作 ===
# 接入 mesh
sudo tailscale up \
  --login-server=https://headscale.example.cn \
  --auth-key=<preauthkey> \
  --accept-routes

# 看状态
tailscale status
tailscale netcheck            # NAT 类型 + DERP 延迟
tailscale ping <peer-ip>      # mesh 层 ping（不走 OS ICMP）

# 重连
sudo tailscale down && sudo tailscale up --login-server=... --accept-routes

九、踩坑总结
#

坑表现根因修复
CIDR 冲突部分端口连不通mesh 路由抢占 docker bridge只 advertise VPC CIDR
删节点 peer 不同步TCP 单向通信Headscale 已知 bug用 ACL 回收，不删节点
用户名必须含 @Headscale 启动失败0.26+ 版本要求用 email 格式作为用户名
K8s Operator 不支持 HA单 Pod 挂就断StatefulSet 单副本固化自管 Deployment，不用 Operator
MagicDNS 不发现 K8s Serviceservice 名不可达设计如此Route53 PHZ 或用 IP
WireGuard UDP 在中国被劣化直连失败走 DERP运营商 QoS中国侧建本地 DERP
控制面单点新连接建不起单实例无 HALitestream + 冷备镜像
客户端 exit-node 风险全流量走 VPN误开 exit-node只 --accept-routes
ACL file 模式 UI 不可编辑Headplane 无法改 ACL设计如此选 file（git 友好）或 database 模式
内核 6.2+ 性能优化吞吐受限老内核缺 GRO仅对 Subnet Router 节点有意义

十、最后
#

Mesh 不是银弹。它解决的是**“如何安全地让人/服务访问内网资源”**这一类问题。它不解决：

  • DB 内的 user 权限和审计（要靠 DB 层 RBAC + 审计日志）
  • K8s RBAC（要靠 K8s 自己的 ACL）
  • 跨境物理延迟（光速极限）
  • 设备自身的安全性（可疑设备应用更高层的设备态势检查）

但它把"我有访问权吗"这一层从分散的 IP 白名单 / SSM Profile / VPN 配置 / 跳板机 SSH key，统一收敛到一个 ACL JSON 文件 + 一次客户端登录。

对于一人运维 + 30 人团队 + 跨云的场景，性价比是其他方案的几倍。

更长远，等团队接入了 SSO（OIDC），PreAuthKey 这一步可以全自动化，开发自助登录 mesh 不再需要运维介入。这是下一阶段的事。
