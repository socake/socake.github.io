元信息

  • 适用规模：5-50 人后端团队，并行 PR 常态 ≥ 3 个
  • 适用云：通用（AWS EKS / 阿里云 ACK / 自建 K8s 都适用）
  • 运维负担：一次性接入 1-2 周；新服务进白名单 0.5 天
  • 月成本：增量 ≈ 现有 QA 成本的 30%（一般 3-5 个并行 PR Pod）
  • 最后验证：2026-04-30，业务服务 A 已端到端跑通真实代码改动

适用场景
#

满足以下任意两条，建议按本 Playbook 推进：

  • 团队同时跑 ≥ 3 个 feature 分支，QA 环境长期被某个 PR 占着，其他人测试拿不到稳定结果
  • 已经尝试过"每个 PR 一套 namespace + 一个域名 + 一张证书"，但维护清理跟不上，半数 namespace 长期 idle
  • QA 环境共享导致回归测试结果"看心情"，开发者把"先在我本地跑过"作为唯一信任源
  • 主干分支保护开了，但 PR 合并前根本没有真实部署过的环境可供 QA 验证
  • 想做"灰度评审"——让产品/QA 在 PR 合并前就在真实部署上点一遍——但又不愿为每个分支维护独立域名

不适用场景见文末「局限」一节。

核心问题
#

共享 QA 的真实代价
#

只看资源开销，“一套 QA 环境"省钱。但工程效率上的隐性成本经常被低估：

痛点表现真实代价
互相覆盖A 推一次镜像，B 正在测的功能版本被替换B 的测试结果作废，重新约时间
配置漂移某 PR 临时改了 Nacos 配置，未还原下个 PR 的故障被错误归因到代码
数据污染A 的 PR 写脏了一张测试表C 的接口测试莫名失败，1-2 天才发现是数据问题
排队“你先测，我等会再 push”高峰期一天只能跑 2-3 轮真机验证

并发 PR 数上升时上述代价成指数增长。一个 5 人后端组同时跑 4 个 PR，每周因共享环境产生的"作废测试"和"误归因排查"按 30 分钟/次估算，单周浪费约 8-10 人时。

想要的形态
#

理想方案具备四个属性：

  1. PR 一推送就有对应的环境，不需要开发者手动申请
  2. 不互相覆盖，每个 PR 有自己的独立 Pod
  3. 入口侧不引入额外的域名/证书运维负担——QA 域名继续用，路由侧自动分流
  4. 自动清理，PR 关闭即销毁，没有人手工 GC 的负担

下面对比三种候选实现，再给出推荐方案。

方案对比
#

方案 A：共享一套 QA 环境
#

形态：一个 namespace、一份 Deployment，所有人推镜像都更新这一份。

适用：单分支主干模型、并行 PR < 2 的小组、对测试结果时效要求低的服务。

淘汰理由：本 Playbook 关心的是并行开发场景，方案 A 正是该场景下的痛点来源。

方案 B：每个 PR 一套完整环境（含独立域名/Ingress）
#

形态：CI 为每个 PR 创建独立 namespace、独立 Deployment、独立 Service，并申请独立域名（如 pr-128.qa.example.cn）+ 独立 TLS 证书 + 独立 Ingress 规则。

适用：

  • 测试团队需要把每个 PR 当作独立的预发环境（含浏览器分享 URL 给产品验收）
  • 已有自助域名平台 + ACME 自动签证书 + Ingress controller 支持动态规则

淘汰理由：

  • 域名/证书运维负担重。三方 DNS 提供商（Route53/阿里云）批量创建解析记录、ACME challenge、Ingress 规则同步，每个环节都可能成为故障点
  • 配置量大。Ingress 规则、Cert-Manager Issuer、可能还要外部 CDN 缓存策略，每个 PR 都要复制一遍
  • 清理麻烦。namespace 要清、DNS 记录要清、证书要清、CDN 缓存策略要清，链路一长就有泄漏

方案 C：PR Pod + X-env header 路由 + 自动清理（推荐）
#

形态：

  • CI 在独立 namespace 拉起 PR Pod，但入口域名继续复用 QA 域名
  • 入口处的 HTTPRoute（Gateway API）按 X-env header 把流量切到对应 PR Pod
  • 三层清理保障：PR 关闭触发清理、24h cron 兜底、ArgoCD ApplicationSet 自动 GC

核心收益：

  • 路由层 0 增量配置：QA 域名/证书/CDN 缓存策略全部复用
  • PR Pod 完全独立：每个 PR 有独立 Deployment + Service，互不打架
  • 开发者使用零摩擦：浏览器装 ModHeader 插件，加一个 X-env: pr-{env_id} header 就接入 PR 环境

后续章节围绕方案 C 展开。

推荐架构
#

flowchart LR
    Dev([开发者推 feature 分支]) -->|webhook| CI[CI / 云效 Flow]
    CI -->|deploy.py 渲染 overlay| Git[(GitOps 仓库)]
    Git -->|ApplicationSet matrix| Argo[ArgoCD]
    Argo -->|kubectl apply| K8s[(独立 namespace<br/>PR Pod + Service)]

    User([测试人员/QA])
    User -->|X-env: pr-feat-channel| Gateway[Istio Gateway<br/>QA 域名不变]
    Gateway -->|HTTPRoute header match| Pod1[PR Pod #128]
    Gateway -->|无 header| Pod0[Base Pod]

    Cron[24h cron 清理] --> Argo
    Closed([PR 关闭/MR merge]) --> Argo
    AppSet[ApplicationSet GC] --> Argo

关键决策点
#

1. 路由层不引入新域名

入口 Gateway 上挂 HTTPRoute，header X-env=pr-{env_id} 命中时把流量打到 PR Service，否则走 base Service。这是整套方案最大的简化点：每多一个 PR，路由层的增量是一个 HTTPRoute 资源（kustomize patch 自动生成），不是一套域名/证书/Ingress。

2. PR Pod 用独立 namespace 还是同 namespace 加前缀？

选择同 namespace + namePrefix pr-{env_id}-。理由：

  • 同 namespace 下 ServiceAccount / Secret / ConfigMap 复用 base 的，不需要为每个 PR 重新拷贝（IRSA / kms 配置一份就够）
  • ApplicationSet matrix generator 沿着 git 目录扫，独立目录已经隔离了 GitOps 资源
  • namespace 多了对监控、日志、限流策略都是负担

3. 入口域名复用：CDN 转发 X-env header

如果入口前还有 CDN（CloudFront / 阿里云 CDN），默认 cache policy 不会转发自定义 header，PR 流量会被错误地命中 base Pod 的缓存。需要为接入 PR 隔离的域名换上自定义 cache policy，把 X-env 加进 whitelist。

数据流
#

  1. 开发者推 feat-channel-backend 分支 → CI 触发
  2. deploy.py 把 PR overlay 写到 gitops/clusters/us-qa/applications/{project}-pr/{env_id}-{service}/
  3. ArgoCD ApplicationSet matrix generator 扫描 git 目录，自动生成 Application
  4. ArgoCD sync → Kustomize build → kubectl apply → PR Pod 起来
  5. 测试人员浏览器 ModHeader 加 X-env: pr-feat-channel-backend，请求被 HTTPRoute 切到 PR Pod
  6. PR 关闭 / 24h cron / 容量超限三种触发任意一种 → 删除 git 目录 → ApplicationSet 自动 GC

实施步骤
#

1. 准备 GitOps 目录约定
#

gitops/
├── base/{project}/{service}/                           # base 共享模板
└── clusters/us-qa/applications/
    ├── {project}/{service}/                            # 普通 QA overlay
    └── {project}-pr/{env_id}-{service}/                # PR overlay（动态生成）
        ├── kustomization.yaml
        └── httproute.yaml

PR overlay 用独立的 {project}-pr 目录，配套独立的 ApplicationSet（matrix 只扫这个目录）。这样 PR Pod 的生灭不影响普通 QA 应用，scan/sync 也独立，定位故障时不会互相干扰。

2. ApplicationSet 配置
#

apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: qa-pr-envs
  namespace: argocd
spec:
  generators:
    - matrix:
        generators:
          - clusters:
              selector:
                matchLabels:
                  env: us-qa
          - git:
              repoURL: ssh://git@git.example.cn/infra/gitops.git
              revision: HEAD
              directories:
                - path: clusters/us-qa/applications/*-pr/*
  template:
    metadata:
      name: '{{path[3]}}-{{path.basename}}'    # e.g. service-foo-pr-feat-channel-backend
    spec:
      destination:
        server: '{{server}}'
        namespace: '{{path[2]}}'                # 与 base 同 namespace
      source:
        repoURL: ssh://git@git.example.cn/infra/gitops.git
        path: '{{path}}'
      syncPolicy:
        automated:
          prune: true
          selfHeal: true

ApplicationSet template 的 app 名前缀建议固定（如 service-foo-pr-），下游清理脚本按前缀匹配，逻辑简单。

3. PR overlay 模板（kustomize）
#

下面是 deploy.py 渲染出来的 overlay 模板核心部分。

# kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../../../../base/{project}/{service}
  - httproute.yaml                  # 必须显式列出，否则 ArgoCD 不应用

namePrefix: "pr-{env_id}-"

labels:
  - includeSelectors: true
    pairs:
      version: "pr-{env_id}"
  - includeSelectors: false
    pairs:
      example.dev/env: qa-pr

images:
  - name: {project}/{service}
    newTag: "{commit_id}-{timestamp}"

patches:
  # 关键 patch：把 app 标签从 base 改为 pr-{env_id}-{base_app}
  # 让 PR Pod 完全脱离 base Service 的 selector
  - target:
      kind: Deployment             # 不写 name，避免硬编码
    patch: |-
      - op: replace
        path: /spec/selector/matchLabels/app
        value: "pr-{env_id}-{base_app}"
      - op: replace
        path: /spec/template/metadata/labels/app
        value: "pr-{env_id}-{base_app}"
  - target:
      kind: Service
    patch: |-
      - op: replace
        path: /spec/selector/app
        value: "pr-{env_id}-{base_app}"

# httproute.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: backend-pr-route       # namePrefix 会加 pr-{env_id}-
spec:
  parentRefs:
    - name: qa-gateway
      namespace: gateway-system
  hostnames:
    - "qa.example.cn"
  rules:
    - matches:
        - headers:
            - name: X-env
              value: "pr-{env_id}"
          path:
            type: PathPrefix
            value: /api
      backendRefs:
        - name: {service}        # kustomize 自动加前缀指向 PR Service
          port: 8080

HTTPRoute 优先级注意：Gateway API 默认 header match 优先级高于无 header 的兜底规则，PR header 命中时不会被 base 抢走。

4. CI 触发与 deploy.py
#

CI 任务在 feature 分支 push 时触发：

python3 gitops/scripts/deploy.py \
    --action create-pr \
    --project service-foo \
    --service backend \
    --branch ${BRANCH} \
    --env-id $(echo "${BRANCH}" | tr '[:upper:]_/' '[:lower:]--') \
    --commit ${COMMIT_ID}

deploy.py 做四件事：

  1. 校验 ({project}, {service}) 在 PR 白名单内
  2. 生成 PR env_id（slug 化分支名）
  3. 渲染 overlay 到 clusters/us-qa/applications/{project}-pr/{env_id}-{service}/
  4. git push GitOps 仓库，ArgoCD 自然 sync

5. 数据库怎么处理
#

最实用的做法是共享 base QA 数据库 + 写入侧由 PR_ENV_ID 隔离。

PR Pod 启动时注入环境变量 PR_ENV_ID=pr-{env_id}。业务代码读取后，在写入路径加前缀或额外字段：

资源类型隔离手段
关系型表（项目/对话/消息）写入侧加 env_tag 字段，按 env_tag = PR_ENV_ID 过滤读
Object Storage / S3key 前缀 pr/{env_id}/...
Redis / Valkeykey 前缀 pr:{env_id}:...
MQ topic多服务联调时用 per-PR topic 前缀（单服务可不做）

不推荐"每个 PR 一个独立数据库”——成本和数据准备复杂度都太高，并且 PR 关闭时还要清。共享库 + 字段隔离对绝大多数场景够用。重要测试数据通过快照恢复或 seed 脚本填，不靠 PR 之间共享。

6. 三层清理保障
#

任何"自动起环境"系统的真正难点是清理。漏清是常态，要靠多层兜底。

第一层：PR 关闭/MR merge 触发

云效 Flow / GitHub Actions 监听 MR merge / close 事件，调 deploy.py --action delete-pr，删 git 目录。ApplicationSet matrix generator 扫不到目录，自动 GC 对应的 Application 和资源。

注意 MR merge 触发时，CI 上下文里的 CI_COMMIT_REF_NAME 是目标分支（main），不是 source 分支。从 webhook payload triggerInfo.msgData.object_attributes.source_branch 解析才能拿到 PR 自己的分支名。

第二层：24h cron 兜底

凌晨 02:00 跑 pr-env-cleanup-cron.py，扫 GitOps 仓库下所有 *-pr/* 目录：

# 伪代码
for overlay in glob("clusters/*/applications/*-pr/*/"):
    branch = parse_branch_from_path(overlay)
    if not git_branch_exists(branch):
        delete_overlay(overlay)
        continue
    if last_modified(overlay) > 7_days:
        notify_owner(branch)
        if last_modified(overlay) > 14_days:
            delete_overlay(overlay)

兜住 webhook 丢失、人工删分支没触发清理等场景。TTL 过期前先发通知给分支 owner，避免误清。

第三层：容量水位 / ApplicationSet 兜底

容量水位告警阈值（PR Pod 数 > N，或 namespace 内存使用 > X%）触发，按 lastSyncedAt 升序删最久未更新的若干个。同时 ApplicationSet 本身的 prune: true 在 git 目录消失后会自动 GC Application 和 K8s 资源。

三层叠加保证"git 目录消失 = K8s 资源消失 = 计费消失"。

踩过的坑
#

坑 1：includeSelectors=true 不够，base Service 反向把 PR Pod 选进 endpoints
#

现象：PR Pod 部署完毕，PR_ENV_ID 标识却泄到了非 PR 请求。base 域名的接口偶发返回带 PR 标识的数据。kubectl 看 endpoints：

$ kubectl get endpoints backend -n qa
NAME      ENDPOINTS                                              AGE
backend   10.x.y.10:8080, 10.x.y.11:8080, 10.x.y.99:8080   1d
                                            ^^^^^^^^^^^^
                                         这个是 PR Pod IP

base Service 3 个 endpoints 里有 1 个是 PR Pod。流量按 endpoint 数比例命中 PR Pod，约 33% 概率。

根因：Kustomize 的 commonLabels / labels.includeSelectors=true 只让 PR Service 的 selector 加上 version=pr-{env_id}，但 PR Pod 还带着 app={base} 这个 label。base Service 的 selector: app={base} 反向把 PR Pod 选进了自己的 endpoints。

隔离名存实亡。

修复：PR overlay 显式 patch deployment 的 spec.selector.matchLabels.app、spec.template.metadata.labels.app、Service 的 spec.selector.app 全部替换为 pr-{env_id}-{base_app}。PR Pod 的 app label 不再等于 base 的 app，base Service selector 选不到 PR Pod。

patches:
  - target: { kind: Deployment }
    patch: |-
      - op: replace
        path: /spec/selector/matchLabels/app
        value: "pr-{env_id}-{base_app}"
      - op: replace
        path: /spec/template/metadata/labels/app
        value: "pr-{env_id}-{base_app}"
  - target: { kind: Service }
    patch: |-
      - op: replace
        path: /spec/selector/app
        value: "pr-{env_id}-{base_app}"

验证：

kubectl get endpoints {base-svc} -n {ns} -o wide
# 期望 IP 列表里没有 PR Pod IP

通用结论：用 commonLabels / includeSelectors 做隔离，永远要反向验证 base Service 的 endpoints。Kustomize 的"自动加 label"只在它自己生成的资源上生效，不会反向修改 base Service 的 selector。隔离的判断标准不是"我的资源带了什么 label"，而是"别人的 selector 选不选得到我"。

坑 2：patch target 写死 name 字段，跨项目失效
#

现象：同一份 PR overlay 模板，业务服务 A 工作良好，业务服务 B 部署完 patch 完全没生效。

根因：base Deployment 的 metadata.name 在不同项目下不一致：

  • base/service-a/backend/deployment.yaml → name: backend
  • base/service-b/backend/deployment.yaml → name: service-b-backend

通用模板里 patch target 写 name: {service}，模板渲染后业务服务 B 这条 patch 找不到目标，静默失败。Kustomize 的"patch target 找不到"默认不报错。

修复：每个 overlay 只有一个 Deployment / Service 时，patch target 只写 kind，不指定 name：

patches:
  - target:
      kind: Deployment      # 不写 name
    patch: |- ...

通用结论：Kustomize patch target 选择器要"刚好够用"。范围太宽会误伤同 kind 的其他资源；范围太窄写死 name 会跨场景失效。overlay 模板覆盖多个项目时，目标资源的 name 经常不一致，name 字段是首要排查项。

坑 3：CloudFront 默认不转发自定义 header，PR 流量被缓存命中 base Pod
#

现象：测试人员加上 X-env header 后，第一次请求路由正确，后续请求又回到 base Pod 的响应。直接 curl 集群内 Service 没问题，从 CloudFront 域名进就不对。

根因：CloudFront 的默认 cache policy Managed-CachingOptimized 不会把自定义 header 加入缓存键，也不一定向源转发。两个后果：

  1. 边缘节点用同一个缓存键服务带/不带 X-env 的请求，先命中谁就给谁的响应
  2. 即使穿透到源，CloudFront 也可能不转发 X-env，源站的 HTTPRoute 收不到 header，按兜底规则走 base

修复：为接入 PR 隔离的域名建独立 cache policy，把 X-env 加进 header whitelist（既参与缓存键也向源转发）：

aws cloudfront create-cache-policy --cache-policy-config '{
  "Name": "pr-isolation-cache",
  "DefaultTTL": 0,
  "MinTTL": 0,
  "MaxTTL": 0,
  "ParametersInCacheKeyAndForwardedToOrigin": {
    "EnableAcceptEncodingGzip": true,
    "HeadersConfig": {
      "HeaderBehavior": "whitelist",
      "Headers": { "Quantity": 1, "Items": ["X-env"] }
    },
    "QueryStringsConfig": { "QueryStringBehavior": "all" },
    "CookiesConfig": { "CookieBehavior": "none" }
  }
}'

CDN 默认策略对自定义路由不友好，几乎是行业通病——Cloudflare、阿里云 CDN 各自有类似坑。通用结论：任何依赖 header 路由的方案，都要在 CDN 层显式声明该 header 参与缓存键和转发；环境验证不能只在集群内 curl，必须从最外层公网域名做端到端验证。

坑 4：异步消费者类服务做 PR 隔离反而更糟
#

现象：把内部异步消费者（如 agent / dispatch 这类纯消费 RabbitMQ 的服务）加进 PR 白名单后，开发者反馈"部署后还是旧代码处理消息"。

根因：这类服务没有 ingress 入口，外部 X-env header 路由进不来；PR Pod 与 base 共享同一个 RabbitMQ consumer group → 消息按概率被 PR Pod 截走，PR Pod 用旧 image / 旧代码处理 → base 的请求方看到的就是"旧代码的结果"。比 base 单跑还糟。

修复：从白名单移除这类服务。如果一定要做 PR 隔离，前置工作有两项：

  1. backend 调下游服务时透传 X-env header
  2. PR Pod 的 consumer group 加 pr-{env_id} 前缀，与 base 隔离

两者满足前不应开 PR 隔离。

通用结论：基于 header 的路由方案天然只覆盖"同步入口流量"。异步消费者、定时任务、scheduler 这类背景作业不在路由覆盖范围内，硬接进 PR 隔离体系反而引入"按概率截消息"的隐性故障。把适用边界写进白名单注释里。

坑 5：PR Pod 启动看似失败但实际部署成功
#

现象：deploy.py 执行后报 ArgoCD API HTTP 403 permission denied，CI 标红，开发者以为部署挂了立刻回滚。

根因：ApplicationSet generator 检测到新目录，到生成 Application 之间有一个 reconcile 间隔（默认 ~3 分钟）。这个窗口里 ArgoCD 还查不到对应 Application，RBAC 默认 deny → HTTP 403。但只要 ApplicationSet reconcile 一次，Application 就会被创建并 automated sync。

修复：deploy.py 的 sync/wait 步骤遇 403 时不要立即 fail，改为最多 60 秒的轮询：

def wait_for_app(name, timeout=60):
    deadline = time.time() + timeout
    while time.time() < deadline:
        r = argocd_get_app(name)
        if r.status_code == 200:
            return r.json()
        if r.status_code == 403:
            time.sleep(5)
            continue
        r.raise_for_status()
    raise TimeoutError(f"Application {name} not visible after {timeout}s")

通用结论：使用 ApplicationSet 时，“应用还没被 generator 创建出来"和"应用部署失败"看起来都是 4xx/5xx，但语义完全不同。涉及 ApplicationSet 的自动化脚本必须显式处理 generator reconcile 窗口。

衡量指标
#

业务服务 A 是首个上线的服务，统计 2 周内 5 名开发者实际使用的数据：

指标Before（共享 QA）After（PR 隔离）
并发可测 PR 数1（互斥占用）≥ 5（实测峰值 7）
单 PR 起环境耗时-（手动协调）镜像构建 + 5 分钟（ArgoCD reconcile + Pod ready）
周 QA 测试冲突上报次数4-6 次0
每 PR 数据污染相关 bug 误报1-2 次/周0
PR 环境月度增量成本-现有 QA 成本的 ~30%（3-5 个常驻 PR Pod）
工程师"等环境"等待时间平均 35 分钟/PR0 分钟（push 即起）

定性变化：

  • 产品/QA 在 PR merge 前就能在真实部署上 review，“发版前的 PR 评审会"取消，改为异步在 PR 上签字
  • 主干分支保护可以更严格——之前担心"严格保护后调试链路太长”，现在 PR 环境可以承担调试角色
  • DevOps 不再处理"QA 环境被 X 占着” tickets

局限
#

适用边界要清楚：

  1. 暂仅一个服务全量上线。业务服务 A 跑通了端到端验证，业务服务 B 同步上线但仍在打磨清理脚本。其余十多个服务进白名单需要逐个评估，工作量约 0.5 天/服务。
  2. 不适合内部异步消费者类服务（agent / dispatch / cronjob 等）。原因见坑 4。
  3. 不适合多服务联调场景下的"链式 PR"——A 服务的 PR 调 B 服务的 PR，需要 B 服务也支持 X-env 透传。当前 backend 没有透传 header 给下游，多服务联调的 PR 隔离还需后续工作。
  4. 数据隔离是"轻量"级别。共享库 + 字段隔离对功能验证够用，对压测、灾备演练、数据迁移类 PR 不适用——这些场景需要独立库或 snapshot。
  5. 前端 PR 环境绕过 K8s。前端走 S3 + CDN，路由层不在集群内的 HTTPRoute 上。前端要做 PR 环境得借助 Lambda@Edge 改 origin，复杂度独立。
  6. 依赖 Gateway API / Istio HTTPRoute。如果入口是裸 nginx-ingress，header match + 多 backend 的支持要看具体 controller 版本，可能需要 ingress annotation 兜路由。

后续演进方向
#

按优先级：

  1. 扩展白名单到主流后端服务。先扩使用频度最高的几个内部服务（一般是企业内 OA 类后端 / 前端、API 网关、relay 等）。
  2. Header 透传方案落地：backend 调下游服务时透传 X-env，多服务联调成为可能。这是后续把 agent / dispatch 进白名单的前置条件。
  3. per-PR consumer group 前缀：异步消费者类服务进白名单的另一个前置条件。
  4. deploy.py 加 --action regenerate-all：模板演进时一键重写所有存量 PR overlay，避免手工批量迁移。
  5. PR 环境数据库 snapshot：高风险 schema PR 启动时从 base 库 snapshot 一份，PR 关闭即删。
  6. 性能测试也走 PR 环境：给 PR 标 perf-test label 时拉起独立 perf 实例 + 流量分流，避免压测影响其他 PR。
  7. Lambda@Edge 改 origin 实现前端 PR 环境（绕开 S3 默认 origin）。

最后验证：2026-04-30。栈版本：Kubernetes 1.30 / Istio 1.22 / Gateway API v1 / ArgoCD 2.12 / Kustomize 5.x。
本 Playbook 描述的是已上线 ~2 周的状态，超过 12 个月后请重新核实工具版本和 Gateway API 字段约定。
