元信息

  • 适用规模：10-200 人团队，5+ 条主干流水线、3+ 个共享数据库
  • 适用云：通用（云效 Flow / GitHub Actions / Jenkins / GitLab CI 思路一致）
  • 运维负担：一次性接入 1-2 周，每条新流水线 0.5-1 小时
  • 月成本：增量 ≈ $0（构建机已有，只是多跑一个 step）
  • 最后验证：2026-04-30，5 条主流水线两周零误阻、拦下两次破坏性 DDL

适用场景
#

满足以下任意两条，建议按本 Playbook 推进：

  • 过去 12 个月发生过至少一次"不兼容 DDL 上 PROD 导致服务回滚"
  • 主干分支合并到 PRE/PROD 之间没有自动的 schema diff 卡口，全靠 DBA 人工审 PR
  • 已经接了 schema diff，但开发反馈"误报多、合并被无意义阻塞"，最后整个 stage 被绕过
  • 多个服务共享一套数据库，单仓库 PR review 看不到跨服务影响
  • 想给 DBA 减负，但又不愿把所有责任压到一个人身上

不适用场景见文末「局限」一节。

核心问题
#

不兼容 DDL 是怎么炸的
#

观察过去两年的真实事故，DDL 把生产打挂的模式高度集中：

模式例子为什么炸
删字段ALTER TABLE orders DROP COLUMN legacy_status旧版本服务还在读这个字段；canary/rolling 期间新旧版本共存
改字段类型ALTER TABLE users MODIFY age VARCHAR(10)ORM 反序列化失败；索引重建期间锁表
加 NOT NULL 没默认值ALTER TABLE x ADD col INT NOT NULL历史行 INSERT 失败；DBA 加列阻塞超长事务
改主键 / 唯一索引ALTER TABLE x DROP PRIMARY KEY, ADD PRIMARY KEY (...)复制延迟暴涨；从库读到不一致状态
RENAME TABLERENAME TABLE old TO new旧版本服务直接 404；缓存 key 失效

这些操作单独看都不算"明显错误"——开发者写 migration 时通常是有理由的，问题出在节奏：DDL 上线和服务上线必须严格匹配先后，但 PR 审核里没人盯。再叠加几个工程现实让事故进一步放大：

  • migration 文件不在主仓库，DBA 单独审；代码 review 看不到 schema 变更
  • canary / 滚动发布期间新旧版本共存 5-15 分钟，这段时间任何"旧版本读不到的字段"都是事故
  • ORM 框架（GORM / SQLAlchemy / Hibernate）在反序列化失败时不一定显式抛错，可能只是某个字段为空，问题被推迟到下游服务才爆
  • 一次破坏性 DDL 在测试环境往往看起来"没事"——因为测试库的数据量小、热点路径覆盖不全

临时方案为什么不够
#

大多数团队的临时方案是DBA 审 migration PR。在 30 人团队下能跑，往 100 人以上扩展时立刻塌：

  • DBA 不熟悉每个服务的代码，只能审 SQL 文本，看不出"删的字段是不是还有人在读"
  • DBA 成为发版瓶颈，开发学会"周五下午提 PR、把 DBA 拖到加班"
  • DBA 偶尔漏审一次，事故必然发生

真正想要的是：把检查机械化，让流水线每次都跑、不漏；同时给开发"提前修"的机会，避免合并前才弹错误导致整窗口浪费。

方案对比
#

方案 A：DBA 人工审核
#

适用：≤ 20 人团队，单 DBA 全权负责，所有 migration 走单独仓库。

淘汰理由（≥ 50 人时）：扩展性差、DBA 知识瓶颈、夜间/周末没人审、无法覆盖跨服务影响。

方案 B：纯 CI 阻塞（合并前必过）
#

在 PR 上跑 schema diff，发现破坏性 DDL 直接 fail，不通过不能合。

适用：migration 仓库独立、CI 误报率 < 5% 的成熟项目。

淘汰理由：

  • 误报多导致开发整体绕过 stage（“先 force merge 再说”）
  • 等待时间长——CI 跑 10 分钟，开发盯着或切走，注意力碎片化
  • 看到 fail 才动手改，整个 PR 窗口浪费一次

方案 C：双 Stage（pre warning + post fail，推荐）
#

把检查拆成两次跑：

Stage时机模式行为
prePR 创建后 / PRE 部署前warning + continueOnError: true + exit 0钉钉/PR 评论提醒；不阻塞
post合并到 PRE 后 / PROD 部署前fail + continueOnError: false + set -e命中破坏性 DDL → 阻塞下一个 stage

关键设计：

  • pre stage 是给开发看的镜子——还没到 PROD，看到问题立刻修，避免"等到合并才发现"的整窗口浪费
  • post stage 是给基础设施留的最后一道闸——pre 被忽略时兜底，必须真阻塞
  • 两个 stage 共用同一个 schema diff 工具和规则，差异只在退出码和通知文案
  • pre stage 的报告标题里明确写"WARNING（不阻塞）"，避免开发者误以为已经 fail 而开始 hot-fix
  • post stage 失败必须把"如何申请豁免 / 找谁拉群"写进消息体，否则发版被卡时 owner 第一反应是急着重跑

适用：本文目标场景。从 5 人到 500 人都能扩展，规则集中维护、通知自动 @ 流水线 owner。实测两周：5 条主流水线（3 条 MySQL + 2 条 PostgreSQL）零误阻、拦下 2 次破坏性 DDL，开发反馈"比之前 DBA 群里追问体验好"。

关键决策点：

  1. 检查源：始终用同一份 baseline（通常是 QA 库的 schema 快照），不要拿 PRE 跟 PROD 对——PRE 本身可能是脏的
  2. 风险等级：DROP / RENAME / 改类型 → 阻塞；ADD / CREATE / 加索引 → warning；纯注释/默认值变更 → 静默通过
  3. owner 制：一个数据库挂在一条主流水线下负责接入，引用方流水线不重复接，避免噪声叠加
  4. ignore-rules 集中维护：所有豁免走 PR 进 GitOps 仓库，不要让各服务自己配，否则规则漂移

实施步骤
#

1. 选 schema diff 工具
#

工具优势劣势
Liquibase多 DB 通用、社区成熟、有 diff 命令XML changelog 学习曲线、对 NOT NULL 默认值检测一般
Flyway轻量、SQL-first、Cloud 版有 drift detection免费版无内置 diff，需自己写
自研脚本完全控制规则、可贴合内部 ORM 习惯维护成本高、新人上手慢
schemahero / atlas声明式、和 GitOps 思路契合改造成本大、对存量仓库不友好

我们选了自研 Python 脚本 + 信息 schema 直查：脚本对 source（QA）和 target（PRE/PROD）连接，跑 INFORMATION_SCHEMA.TABLES / COLUMNS / STATISTICS 比对，输出"缺表 / 缺字段 / 类型不一致 / 索引差异"四类。原因是历史 migration 不在统一仓库，Liquibase 反推 changelog 成本太高，直查 metadata 反而最直接。

工具决策时建议同时回答这几个问题：

  • migration 是不是有统一仓库？没有就别选 Liquibase changelog 模式
  • 跨 DB 引擎（MySQL + PG）要不要一套规则？要的话首选 atlas 或自研
  • 有没有 SaaS 限制（数据出境、连接 IP 白名单）？SaaS 工具往往需要把生产 DB 暴露给第三方

2. 双 stage 集成（云效 Flow 示例）
#

云效 Flow YAML 片段，pre stage（warning）：

stage_schema_check_pre:
  name: schema_check (PRE deploy 前)
  needs: [stage_build]
  jobs:
    job_check_pre:
      runsOn:
        group: ${BUILD_GROUP}
      steps:
        step_check:
          step: Command
          continueOnError: true   # 关键：warning 模式不阻塞
          with:
            run: |
              set +e
              git clone --depth 1 https://example.com/infra/gitops.git
              cd gitops/scripts/db-schema-check
              bash runner-pipeline.sh pre-deploy "${PIPELINE_NAME}" mysql "${DB_NAME}" pre
              # 永远 exit 0，靠钉钉通知
              exit 0

post stage（fail）：

stage_schema_check_post:
  name: schema_check (PROD deploy 前)
  needs: [stage_deploy_pre]
  jobs:
    job_check_post:
      runsOn:
        group: ${BUILD_GROUP}
      steps:
        step_check:
          step: Command
          continueOnError: false   # 关键：必须真阻塞
          with:
            run: |
              set -e
              git clone --depth 1 https://example.com/infra/gitops.git
              cd gitops/scripts/db-schema-check
              bash runner-pipeline.sh post-deploy "${PIPELINE_NAME}" mysql "${DB_NAME}" pre
              # 脚本内部命中阻塞规则会 exit 1

GitHub Actions 思路一致，pre stage 用 continue-on-error: true，post stage 不加。Jenkins 用 unstable vs failure 状态映射。

3. 风险等级配置
#

ignore-rules.yaml 集中管理：

risk_levels:
  block:                   # 命中即 post stage fail
    - DROP_TABLE
    - DROP_COLUMN
    - ALTER_COLUMN_TYPE
    - RENAME_TABLE
    - DROP_INDEX_PRIMARY
  warn:                    # pre/post 都通知，不阻塞
    - ADD_COLUMN_NOT_NULL_NO_DEFAULT
    - ADD_UNIQUE_INDEX
  silent:                  # 完全不报
    - ADD_COLUMN_NULLABLE
    - ADD_INDEX_NORMAL
    - COMMENT_CHANGE

ignore_tables:             # 已知豁免（业务确认）
  business_db:
    - tmp_migration_*
    - audit_log_2024_*

加豁免必须走 PR review，不要让流水线 owner 直接 push——否则规则会被悄悄放宽。

4. 通知通道：owner 第一眼看到什么
#

通知设计是双 Stage 模式的胜负手。Log/钉钉必须满足：

  • 状态首屏可见：PASS / FAIL / WARN 在 log 第一行，钉钉标题带 emoji 状态
  • 下一步动作独立成块：紧跟状态行写"接下来怎么做"（修 migration / 加 ignore-rule / 联系 DBA），不要混在缺表清单里
  • @ 真正的 owner：从流水线 globalParams（如 DINGTALK csv）取手机号，不要硬编码维护者
  • 详情清单放最后：缺表名、字段差异、SQL 片段——重要但不是首屏决策依据

钉钉 markdown 模板：

### Schema Check FAIL  业务服务 A
**结论**：发现 3 处破坏性 DDL，PROD 推进已阻塞

**下一步**：
- 选项 1：在 migration 里补 backfill 逻辑后重提
- 选项 2：评估后加 ignore-rule（PR 走 DBA review）
- 选项 3：找同事 X / 同事 Y 讨论

**详情**：
- DROP_COLUMN: orders.legacy_status
- ALTER_TYPE: users.age (INT → VARCHAR)
...

@13800138000 @13900139000   <!-- 流水线 owner -->

5. owner 视角验证（强烈建议）
#

工具上线前先模拟一次盲读：假装自己是周一被 @ 醒的值班开发，点开钉钉/log，问自己——

  • 三秒内看到结论了吗？
  • 知道下一步该做什么了吗？
  • 还需要往下翻才能找到关键信息吗？

如果哪一项不达标，重排 log 顺序、调通知模板。这一步不做，后面所有"技术上跑通"都是纸老虎。

实操中我们还加了一条硬规定：新流水线接入后强制故意触发一次 fail——找一张测试库的废表手动 DROP，看 post stage 是不是真红、钉钉有没有 @ 对人、log 首屏是不是结论。这一步走完才算上线完成。

踩过的坑
#

坑 1：纸老虎陷阱——技术上跑通，语义上没拦
#

现象：5 条流水线一次性接入，post stage 跑了 3 周，run 全部绿色。某次主动抽查发现一条流水线累计 25 张缺表从未触发阻塞，差点带到 PROD。

根因：批量推送时图省事，post stage 直接复用了 pre stage 的 continueOnError: true + exit 0。从外观看 stage 名字叫 schema_check_post、状态绿色，谁都以为已经在守护——实际 stage 永远不会 fail。

修复：post stage 强制 continueOnError: false + set -e，runner 脚本明确区分 pre-deploy / post-deploy 两个模式，post 命中破坏性 DDL 必 exit 1。同时在 PR review checklist 里加一条：“新接入流水线必须看一次失败 case 截图”。

通用结论：拦截类工具上线后做一次反向 review——故意制造一次失败，看 stage 是不是真的红了、下游是不是真的停了。“看起来在拦但实际没拦"比"完全没装"还危险，因为它制造了"已被守护"的错觉。

坑 2：云效 step envs 字段被静默忽略
#

现象：把数据库连接信息写在 step envs: 里，run 跑起来发现 ${DB_HOST} 是空字符串，脚本连了个错的库报"connection refused”。

根因：云效 Flow YAML 的 step 级别 envs: 字段不报错但实际不注入环境变量——文档里写了，但没注意。job 级别和 stage 级别的 env 也有类似坑。

修复：所有变量在 run: 块内 export，或者用 ${VAR} 占位符让云效在渲染期替换：

steps:
  step_check:
    step: Command
    with:
      run: |
        set -e
        export DB_HOST="${DB_HOST_PRE}"   # 来自变量组
        export DB_PASS="${PG_PASS_PRE}"
        bash runner.sh

通用结论：CI 工具的非显然 YAML 约束都是"不报错但行为错误"型——比变量名拼错更难排查。新写流水线先用 env | grep -i db 打一行 debug 输出，确认变量真的注入。

坑 3：伪阳性误伤——同表多次 ALTER 被合并识别为高危
#

现象：开发在一个 migration 文件里先 ADD COLUMN x INT NULL，后面 ALTER COLUMN x INT NOT NULL（先加列再补 backfill 再加约束）。schema diff 直接看 source vs target 终态，识别成"加了 NOT NULL 字段没默认值"，被 post stage 阻塞。

根因：schema diff 工具看的是 INFORMATION_SCHEMA 终态，不是 migration 路径。开发的中间步骤已经把数据填好了，但工具不知道。

修复：

  • 短期：在 ignore-rules.yaml 加白名单，针对该字段豁免一次
  • 中期：runner 增加"扫 migration 文件"模式，识别同 PR 内的 backfill 语义
  • 长期：要求所有破坏性 DDL 在 PR 描述里附带 migration plan: <link>，由 DBA 二次确认后加 ignore-rule

通用结论：schema diff 的精度永远不可能 100%。设计时要预留人工豁免通道，并且豁免必须留痕（PR record + reviewer + 过期日期），否则一年后没人记得为什么放过去。

坑 4：多服务共享一个库，A 服务的 check 拦不住 B 服务的破坏
#

现象：业务服务 A 和业务服务 B 共用一个 MySQL 库。A 仓库接了 schema_check，B 仓库没接。B 的开发提了一次 DROP COLUMN，B 流水线一路绿，到 PROD 才发现 A 的旧版本读这个字段崩了。

根因：schema check 挂在仓库流水线上，不是挂在数据库上。共享库的破坏性 DDL 来自任何一个仓库都能逃逸。

修复：

  • 主库 owner 制：每个共享库指定一个"主流水线"接 schema_check，其他服务的流水线在 PR description 里声明使用了哪个共享库，主流水线 owner 在跨服务 DDL 时主动通知
  • runner 脚本在缺表 / 多表清单里附加"该表在哪些服务中被引用"——通过 grep 业务代码 import 关系生成
  • 中期：把 schema_check 从仓库流水线下沉到独立的"DB 守护流水线"，定时跑 + 各仓库 PR 触发，跨仓库共享视角

通用结论：检查的边界要和资源的边界一致，不是和仓库的边界一致。共享资源（DB / 共享 Kafka topic / 共享缓存 key 空间）必须有专门的守护通道。

坑 5：钉钉 @ 错人
#

现象：上线第一周，schema check 阻塞了 7 次，全部 @ 维护者一个人。维护者周末被 @ 醒后才发现自己根本不懂 B 服务。

根因：runner 脚本里硬编码了通知手机号——“先跑通再说"思路下的临时代码忘了改。

修复：从流水线 globalParams.DINGTALK（csv 手机号列表）读取，runner 用 ${DINGTALK:-${DING_AT_MOBILES:-}} 兜底。每条流水线 owner 自己维护这个变量。

通用结论：通知 @ 谁 = 真正能处理的人，不是默认管理员。复用流水线已有的 owner 配置，不要搞新一套。

衡量指标
#

接入两周后的 before/after：

指标接入前（过去 90 天平均）接入后（最近 14 天）变化
月度 DDL 引发的 PROD 事故1.3 次0 次-100%
DBA 人工审 migration PR 工时8 小时/周2 小时/周-75%
开发等待 schema 评审平均时长6.2 小时1.1 小时-82%
pre stage 触发率（每条 PR）-23% PR 触发 warning信号比预期高
post stage 真阻塞次数-2 次真有牙
误报率（warning 但实际安全）-~12%加 ignore-rule 后降到 4%

定性变化：

  • DBA 不再周五加班审 PR，节奏从"被开发拖着审"变成"主动看周报里高风险 DDL 列表”
  • 开发把"先看 pre stage 输出"作为提 PR 的标准动作，部分团队甚至把 pre stage 报告链接贴进 PR description
  • 一次跨服务 DDL 沟通从"发钉钉群讨论 2 天"变成"看主库 owner 钉钉提示直接回复"
  • 新人 onboarding 时间缩短——以前要背"哪些 DDL 不能上"的口头规则，现在只要看一次失败 case 就懂

需要警惕的反指标：

  • 如果 ignore-rules 一周新增 > 5 条，说明规则太严或工具误报多，不是"开发太皮"
  • 如果 post stage fail 率 = 0% 持续超过一个月，要么真的没人写破坏性 DDL（罕见），要么规则失效了——主动跑一次故意失败的回归

局限
#

本 Playbook 不解决以下问题：

  • 数据迁移逻辑错误：UPDATE x SET status='new' WHERE old_status='legacy' 写错了过滤条件，schema diff 看不出来
  • 跨表关联破坏：删 A 表的字段虽然兼容了 A 的服务，但 B 表的外键引用断了
  • DML 风暴：大 batch UPDATE/DELETE 锁表，schema 维度无法识别
  • migration 路径不可逆：DROP 之后才发现需要回滚，工具不会帮你存 backup
  • schema diff 工具本身的盲区：触发器、视图、存储过程的支持各家工具差异巨大，需自己测试

这些场景需要补充：DML review checklist / 数据迁移测试集 / 影子库回放 / 自动化备份。

后续演进方向
#

未来 6-12 个月计划：

  1. 流水线集成数据迁移测试——每次 migration 在影子库（PROD 备份的 1% 抽样）上 dry-run，验证执行时间和锁竞争
  2. canary 自动回滚——PROD 部署后 5 分钟内监控 ORM 错误率，超阈值自动 rollback migration
  3. 跨集群同步检查——业务跨多 region 部署时，schema 漂移会让某 region 静默挂掉。在 post stage 加全 region 一致性校验
  4. migration timeline 可视化——每个共享库一个 dashboard，展示过去 30 天所有 DDL + 命中规则 + 影响服务，给 DBA 月度复盘用
  5. 从仓库 stage 下沉到独立守护流水线——共享库 owner 模式扩展到所有 30+ DB，定时跑 + 仓库 PR 触发双源信号

最后验证：2026-04-30，云效 Flow YAML 2026-04 / Liquibase 4.27 / Flyway 10.x / Aurora MySQL 8.0 / Aurora PostgreSQL 15。超过 12 个月后云效 YAML schema 可能演进，请以官方文档为准。
