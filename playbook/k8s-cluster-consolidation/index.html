元信息

  • 适用规模：3-5 个相似业务集群 / 总节点数 30-150
  • 适用云：AWS EKS / 阿里云 ACK / 自建 K8s
  • 运维负担：合并期 1-2 人 × 2-4 周；稳定后单人即可维护
  • 月成本节省：典型案例 200-500 USD / 集群（control plane + 中间件 + 监控冗余）
  • 最后验证：2026-04-30，Karpenter v1.5.0 + EKS 1.31

适用场景
#

满足下面任意三条以上时，集群合并方案适用：

  • 同一业务在不同环境（QA / PRE / Staging / Sandbox）各跑一个集群，架构同构
  • 单集群节点数长期低于 10、平均利用率低于 30%
  • 监控、日志、CSI、Ingress 这类辅助组件在每个集群都重复部署
  • 平台版本升级（K8s 小版本、CNI、CSI、Ingress）每次都要重复做 N 次
  • 团队希望保留环境隔离语义（namespace、网络、中间件），但不再为每个环境付一份基础设施账单

不适用的场景见文末「局限」一节。

核心问题
#

把多集群运维一年以上的团队，普遍会遇到下面三类浪费：

  1. 节点利用率低：每个集群至少要保留若干 system 节点（CoreDNS、metrics-server、CSI、Ingress Controller），加起来动辄 6-8 个节点常驻。但实际业务 Pod 总量并不大。
  2. 重复跑监控与日志：Prometheus、Loki、Promtail、Grafana Agent 在每个集群都开一份，存储与采集 Agent 的 CPU/内存占用是固定开销，跟业务量无关。
  3. 平台升级要做 N 次：升级 EKS 小版本、轮换 CNI、给 Karpenter 加一类 NodePool，每个集群要重复一遍。每次升级都要走一次完整验证，消耗的时间远超技术工作本身。

临时缓解方案常见的有：把测试集群节点缩到极限、Spot 实例兜底、用脚本批量同步 manifest。这些都是单点止血，不解决「集群本身就是冗余」这个结构性问题。

真正想要的是：一套 K8s 控制面 + 共享的辅助组件 + 多个互相隔离的环境命名空间。控制面合并节省固定成本，数据面（业务 Pod、网络、中间件）保留隔离防止污染。

方案对比
#

候选方案适用淘汰理由
维持多集群，单独运维强合规 / 不同业务线 / 跨地域固定成本浪费 + 升级负担线性增长
合并到一个集群，仅 namespace 隔离内部工具 / 单一团队缺乏网络与中间件隔离，跨环境污染概率高
合并到一个集群 + namespace + NodePool taint + NetworkPolicy + 中间件独立多环境同业务 / 测试集群维护略复杂，但隔离强度接近原多集群

候选 1：维持多集群，单独运维
#

这是默认状态。每个环境独立 K8s 集群，独立 control plane、独立 worker、独立中间件。

适用：跨地域容灾、强合规要求（PCI / SOC2 边界）、不同业务线（电商 / 金融 / 内部工具）。

淘汰理由：在「同一业务的多个环境」这个语境下，集群之间的差异只是数据，运行的代码、监控规则、Ingress 规则几乎一致。多集群带来的隔离收益小于运维成本。

候选 2：合并到一个集群，仅 namespace 隔离
#

把所有环境塞进一个集群，每个环境一个 namespace。namespace 本身做权限隔离（RBAC）、资源配额（ResourceQuota）。

适用：内部工具、对隔离强度要求很低的开发集群。

淘汰理由：namespace 默认不做网络隔离，A 命名空间的 Pod 可以随手 curl 到 B 命名空间的 Service。如果业务又共享了 RabbitMQ、Kafka、Redis、RDS，那就只剩「逻辑边界」，跨环境污染只是时间问题。本文「踩过的坑」一节给了一个真实事故。

候选 3：合并到一个集群 + 多层隔离（推荐）
#

合并控制面，但在数据面保留四层隔离：

  • NodePool taint：每个环境独立 NodePool，节点打 taint，业务 Pod 加 toleration，物理上不混跑
  • Namespace：每个环境独立 namespace，配合 RBAC、ResourceQuota
  • NetworkPolicy / Cilium：默认拒绝跨 namespace 流量，按需打开
  • 中间件独立：每个环境独立的 RDS / Kafka / Redis，或者至少在共享实例里用独立 schema / topic / key prefix

适用：同一业务的多个测试环境（QA / PRE / Sandbox / Demo），合并后保留环境语义。

取舍：维护成本比方案 2 略高（多了 NodePool YAML、NetworkPolicy、中间件 endpoint 管理），但隔离强度接近多集群方案。下面的「推荐架构」和「实施步骤」围绕方案 3 展开。

关键决策点：

  • 控制面共享：API Server、Karpenter、ArgoCD、Prometheus、Loki 全部一份。这是合并的主要省钱来源。
  • NodePool 物理隔离：每个环境独立 NodePool + taint，避免一个环境的高负载 Pod 抢占另一个环境的资源。需要时也可以让 system 节点保留独立 NodePool（CoreDNS / Ingress 共享，业务节点隔离）。
  • NetworkPolicy 默认拒绝：合并后跨 namespace 网络默认开放是大坑，必须用 Cilium 或原生 NetworkPolicy 显式 deny，再按服务依赖打开。
  • 中间件不能共享：RDS / RabbitMQ / Kafka / Redis 必须每个环境独立实例，或至少独立 logical schema / topic / vhost / key prefix。这是本文最重要的结论，反面案例见下文「踩过的坑」。
  • 统一 Ingress：合并后只跑一份 Ingress Controller / NLB，TLS 证书改成多 SAN 一次覆盖三个环境域名。

实施步骤
#

下面是一个三环境合并的标准流程，按顺序执行可以做到无中断切换。

步骤 1：评估前置
#

合并前必须把下面四类信息列清楚，缺一个都会在切换日炸一个坑。

  • 隔离边界：每个环境对外暴露什么域名、内部依赖什么中间件、是否有跨环境的 webhook 或 cron
  • 共享资源：是否有「跨集群共享的 EFS / S3 / ECR / Nacos namespace」，合并后还能不能继续共享
  • 辅助组件清单：placeholder、scaler、node-protector、PVC 备份、CronJob，这些往往不在主线业务清单里，最容易漏迁
  • 业务表自增 ID 现状：跨环境的 id 区间有没有重叠，重叠会在中间件共享时触发跨环境污染（详见踩坑 1）

步骤 2：NodePool 设计
#

每个环境一个 NodePool，taint + label 双标记。Karpenter NodePool 示例（仅展示关键字段）：

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: gvisor-pre
spec:
  template:
    metadata:
      labels:
        sandbox.env: pre
        sandbox.gvisor/enabled: "true"
    spec:
      taints:
        - key: sandbox.env
          value: pre
          effect: NoSchedule
      requirements:
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["on-demand"]
  limits:
    cpu: 16
    memory: 64Gi
  disruption:
    consolidationPolicy: WhenEmpty
    consolidateAfter: 10m

业务 Deployment 加对应 toleration + nodeSelector 即可调度到对应 NodePool。

注意一个常见反向陷阱：如果业务 Pod 的 nodeSelector 是「sandbox.gvisor/enabled=true」这种宽泛标签，会同时匹配 qa / pre / ai 三个 NodePool。要么 nodeSelector 写死到 sandbox.env: pre 这个精确值，要么在 nodeAffinity 里加 DoesNotExist 表达式排除其他环境的 label。

步骤 3：命名空间迁移顺序
#

迁移顺序遵循「风险从低到高」原则：

  1. 第一批：开发自助沙箱、Demo 环境这类「炸了也就算了」的 namespace，验证整套流程
  2. 第二批：QA 环境，业务回归走完整链路
  3. 第三批：PRE 环境，迁移前必须有完整回滚预案

每一批迁移完成后留 3-7 天观察期再做下一批。中间任何一批出现稳定性问题，立即停下排查根因，不要赶进度。

步骤 4：中间件独立配置
#

合并后中间件配置变化最大，按下面四条对照检查：

  • RDS：每个环境独立 instance，或独立 schema + 独立账号。共享 instance 时业务表自增 ID 起点要拉到 ≥ 10000000，避免跨环境 ID 撞车
  • Kafka：独立 topic 前缀（pre.、qa.、ai.），独立 consumer group，避免一边 commit offset 影响另一边
  • RabbitMQ：独立 broker 是首选；如必须共享，至少独立 vhost + 独立 exchange + 独立 queue 命名前缀
  • Redis / Valkey：独立 instance 或独立 logical DB；共享 instance 时所有 key 必须带环境前缀，应用代码层强制校验

每个服务的配置文件（Nacos / ConfigMap / env）都要扫一遍，确认没有指向旧环境的中间件 endpoint。

步骤 5：端到端验证 checklist
#

合并不是「Pod 全部 Running」就结束。下面这份清单走完才算合并完成：

  • DNS：每个环境的对外域名解析正确，证书覆盖所有 SAN
  • Ingress：HTTP / gRPC / WebSocket 三类路径都通
  • service mesh / 服务发现：跨 namespace 调用按预期被 NetworkPolicy 拦截或放行
  • 监控：Prometheus 抓到所有 namespace 的指标，Grafana dashboard 区分环境
  • 告警：Alertmanager 路由按 namespace 路由到对应通知渠道，没有把所有告警混进同一个群
  • 日志：Loki / Promtail 抓到所有节点的日志，label 包含 env
  • 业务链路：跑一次完整业务流程（创建 → 操作 → 销毁），不能只看健康检查
  • 辅助组件：placeholder、scaler、CronJob、Webhook 全部确认在新集群运行
  • 中间件：每个环境独立连，跨环境无法访问
  • 回滚预案：旧集群保留控制面（nodegroup 缩到 0），观察期通过后再删

踩过的坑
#

下面三个坑都来自一次真实合并项目，每个坑都对应一个独立的事故。

坑 1：共享 RabbitMQ + 自增 ID 撞车，10 万条脏消息污染老项目
#

现象：合并后某天，QA 环境的内部测试用户反馈，自己 7 个月前的老项目突然多出大量 AI 助手对话记录，里面是其他用户今天的内容。受影响 10 个用户、150 个项目，QA 库 m_realtime_messages 表多出 91,403 行脏数据，3 个相关消息表合计约 10 万行。

根因（四件套同时成立才会触发）：

  1. AI 环境 m_project.id 从 1 独立自增（最大才 274）
  2. QA 环境 m_project.id 老项目大量在 1-274 区间（半年前的项目）
  3. RabbitMQ broker + vhost 两个环境共用
  4. 消费端按 project_id 数字写本地库，没有 dispatch_env 字段过滤跨环境消息

完整链路：AI 后端推 WebSocket 消息到共享 RabbitMQ → QA 和 AI 两边的 dispatch 服务都消费 → 各自按 project_id 数字写本地库 → AI 的 project_id=233 撞上 QA 半年前用户 15 的「小红书」老项目 → 用户在 QA 端打开老项目就看到了今天 AI 用户的对话流。

修复：

  • P0：起独立 RabbitMQ broker（mq.m7g.medium，约 71 USD/月）+ 独立 Valkey（约 11 USD/月）
  • P0：改 Nacos 配置 6 个字段切到新中间件，滚动重启相关服务
  • P1：数据清洗（CTAS 备份 + 临时表中转 + 分批 DELETE）共清掉约 95k 行
  • P1：研发侧加 dispatch_env 字段过滤（防御层）
  • P2：AI 库的业务表自增 ID 起点提到 10000000+

通用结论：

  • 共享中间件 + 自增 ID 是跨环境污染的最隐蔽组合。事故前两边业务都跑得好好的，只在数据层悄悄交叉。
  • 新环境不能从老环境直接复制粘贴配置，必须逐项核对中间件 endpoint 与 ID 起点。
  • 业务表自增 ID 在多环境复用同一个中间件时，必须人为拉开起点（≥ 10000000）。
  • 排查跨环境数据混用，先看自增 ID 区间是否重叠，这条路径比看流量比看 broker 都直接。

坑 2：辅助组件没迁移，扩容机制悄悄失效
#

现象：合并几天后，AI 环境的用户报「沙箱创建超时」。检查发现 AI 环境的 NodePool 一直零节点，新建沙箱请求触发的 Pod 处于 Pending，但没有任何节点扩容。

根因：

  • 合并清单只写了主业务 workload（agent / portal / meter / ui），漏了 placeholder-controller 这个辅助组件
  • placeholder-controller 在新集群的 replicas 是 0，没有保底 Pod 占位
  • DaemonSet 不会触发 Karpenter 扩容（DaemonSet 的 Pod 只跟着节点跑，不创建节点）
  • 同时 Nacos 里 scaler.nodepool_name 字段没改，默认值还指向旧集群的 NodePool 名

修复：把 placeholder-controller 部署到新集群、replicas 改 1；Nacos 里所有指向旧 NodePool 名的字段逐个改过来。

通用结论：

  • 集群合并的盲区不是主业务，而是 placeholder / scaler / protector / Webhook / CronJob 这类「不太显眼但很关键」的辅助组件
  • DaemonSet 单独存在不会触发节点创建。NodePool 必须有非 DaemonSet Pod 兜底（placeholder Deployment）才能从零起节点
  • Nacos / 配置中心的字段要逐字段核对默认值，不是「服务起来了」就够了。默认值很可能还指向旧资源

坑 3：以为有 S3 备份，其实只是本地盘
#

现象：合并过程中要把 Loki 的旧 PVC 数据同步过去。运维以为 Loki 数据有 S3 兜底，先把旧 StatefulSet scale 到 0 再开始迁移。结果 PVC 因为 persistentVolumeClaimRetentionPolicy 自动回收，38 天的监控日志全部丢失。

根因：

  • 没读 Loki 配置就假设有 S3 backend，实际配置里 object_store: filesystem，PVC 是唯一数据源
  • StatefulSet 的 persistentVolumeClaimRetentionPolicy: Delete 默认 + scale 到 0 触发了 PVC 自动回收
  • 操作前没和团队同步「数据风险」，事后才发现

修复：从下游 Grafana 缓存导出部分关键面板数据；后续给 Loki 切到 S3 backend，杜绝单点。

通用结论：

  • 操作有状态服务前，必须先读配置文件确认数据存储后端，绝不能假设
  • 如果是 filesystem backend，PVC 就是唯一数据源，不能 scale 到 0、不能删 PVC
  • 大版本变更前先看 persistentVolumeClaimRetentionPolicy，有 Delete 策略就主动改成 Retain，做完再改回来
  • 不确定数据安全时，先把风险摆出来等确认，再动手

坑 4：共享 EFS 跨 VPC 挂载的「历史债」
#

现象：合并时发现，AI 环境其实早就跨 VPC 挂载了 QA 业务集群的 EFS，节省 EFS 实例的同时，把 sandbox 数据和生产业务数据放在了同一个文件系统里。新一轮 PRE 合并时如果直接复用这个模式，会让生产 EFS 承载更多沙箱数据。

根因：早期为了省事，AI 集群的 NodePool userData 直接 mount 了 QA 业务 EFS 的内网 IP，没人正式 review 过这个决策。

修复：本次合并不继承这个模式，给 sandbox 单独建独立 EFS（85GB，约 7 USD/月），用 rsync 同步数据后切换。

通用结论：

  • 合并前必须把「共享资源全景」画一张图，明确哪些是历史债不该继承
  • 新方案的隔离边界应该向更强的方向走，不应该叠加历史的混用模式
  • 跨 VPC 挂载 EFS 在技术上能跑，但隔离语义会被打破，应该作为例外而不是默认

  • 正向：合并触发了一轮配置审计，多年没人动的 Nacos 字段被重新核对
  • 负向：合并初期注意力大量集中在「Pod 跑起来」，对中间件隔离审计不足，导致了坑 1 的事故
  • 负向：跨环境网络从「物理隔离」降级到「策略隔离」，对 NetworkPolicy 维护质量提出了更高要求

局限
#

下面这些场景不适用集群合并：

  • 生产环境：Prod 永远独立集群。控制面故障的 blast radius 不能扩散到 Prod
  • 跨业务线合并：电商、金融、内部工具不应该塞进同一个集群，权责边界比成本重要
  • 强合规边界：PCI-DSS / SOC2 / HIPAA 等合规要求集群级隔离的场景，不要为了省钱触碰
  • 跨地域：不同地域的集群不能合并，延迟和容灾要求都不允许
  • 业务隔离强度要求 ≥ 5：如果业务方明确要求「不同环境物理不能互通」，多集群是唯一答案
  • 节点数量超过 300 单集群：单集群规模过大时反而要拆，不是合
  • 多团队权责不清：合并后所有团队共享一个集群，权限和故障归属容易扯皮，这点比技术问题更难解决

后续演进方向
#

合并不是终点，下面三条是后续 6-12 个月可以推进的方向：

  1. FinOps 看板：按 namespace + label 拆分集群账单，把「环境成本」可视化到团队，避免合并后大家失去成本敏感度
  2. 跨集群迁移自动化：把这次手工跑的 namespace 迁移流程脚本化（Karpenter NodePool 模板 + Nacos 字段自动改写 + Secret 同步），下次需要再合并时复用
  3. 反向拆分机制：业务规模快速增长时，要有把某个 namespace 单独「分裂」回独立集群的能力。提前把 GitOps 目录结构按 namespace 隔离设计好，分裂时只是把目录搬到新集群仓库
  4. 隔离强度自动审计：每周扫一次「跨 namespace 流量是否被 NetworkPolicy 拦截」「中间件连接来源是否符合预期」，把隐性的污染风险显性化
  5. 合并后冷却期标准化：每次合并后强制 1-2 周观察期，期间禁止其他大变更，避免新增问题与合并问题相互掩盖

最后验证：2026-04-30，Karpenter v1.5.0 + EKS 1.31 + ArgoCD 2.13 + Cilium 1.16

超过 12 个月请重新评估：Karpenter NodePool API 在 v1 之后字段会动；EKS 自带的 NodePool 管理（自托管 Karpenter）在 1.31 起逐步推广，可能改变本方案中 Karpenter 的部署方式。
