元信息

  • 适用规模：10-200 人团队，2-10 个 K8s 集群
  • 适用云：AWS / 阿里云 / 混合云
  • 运维负担：单人可维护，前期合并需 2-3 周专项投入
  • 月成本：约 ¥0（基于既有 Prometheus 栈，仅做整合）
  • 最后验证：2026-04-30，Prometheus 2.54 + Alertmanager 0.27 + prometheus-webhook-dingtalk 2.1.0

适用场景
#

下面的条件满足任意三条以上时，本方案适用：

  • 同时存在两套及以上告警系统（典型组合：Prometheus 自建 + 业务侧 SQL 轮询 + 云厂商 CMS）
  • 告警通知散落在多个钉钉/飞书群，没人能讲清"这条告警从哪来"
  • 历史 silence 规则堆积，没人敢清，也没人知道哪些被静默了
  • 曾经发生过"告警规则失效几十天但没人发现"的事件
  • 新增 AWS / 阿里云 资源告警时，不确定该绑哪个 SNS topic 或 Lambda
  • 老 Lambda 模板硬编码业务名（“RabbitMQ 告警: …"），但实际已被复用到无关告警上

不适用的场景见文末「局限」一节。

核心问题
#

告警体系的常见演化路径是：先有 Prometheus + Alertmanager 跑业务指标，后来某个开发顺手在业务后端写了一套定时 SQL 轮询当作"业务监控”，再后来云厂商的告警没人接管也不敢关，最终形成三套并行运行、互不感知的体系。

实际生产中观察到的问题：

  • 规则总数失控：单云 95 条 Prometheus 规则 + 业务侧 103 条业务监控规则，加 CN 云监控的几十条，谁也讲不清覆盖了什么、漏了什么
  • 重复覆盖：同一个症状（如 5xx 飙升）三套体系都报，钉钉群同一时间收三条不同格式的消息
  • 渠道交叉：新增 Aurora 告警时被绑到 SNS topic rabbitmq-prod-alerts，复用了一个 Lambda，结果该 Lambda 代码里硬编码了 title = "RabbitMQ 告警: ..." 和 Broker: prod-xxx-rabbitmq-ha-v2，Aurora 告警发出去全是 RabbitMQ 字样
  • silence 黑魔法：上线日临时 silence 全部告警时图省事写 alertname=~".+"，把 Watchdog 心跳一并屏蔽，反向告警机制（dead-man’s-switch）失效，整条链路坏了几小时没人发现
  • 规则失效无人发现：某集群 12 条沙箱告警因为 label 不匹配 ruleSelector，75 天没生效过一次，告警里写的范围跟实际生效的不是一回事

这些问题的共同特征是：告警体系不是一次性建好就完事的工程，是有腐化速度的活物。每个新加的资源、每次故障复盘、每次临时 silence，都在往里面堆熵。如果没有定期治理动作，半年后就是失控状态。

真正想要的是一套结构清晰、来源可追溯、改动有迹可循的告警体系：每条告警讲得清"为什么报、谁负责、如何处置"，每条 silence 写得清"屏蔽什么、什么时候到期、是否豁免心跳"，新增告警时有明确的接入路径不会误绑老 Lambda；每周自动盘点一次哪些规则在 fire、哪些在 silence、哪些 30 天没动过，腐化苗头能在变成事故之前被发现。

方案对比
#

方案 A：维持现状，每次故障再补
#

继续让三套体系各自运行，谁负责的部分谁修，事故复盘时再补规则。

  • 适用：团队规模 < 5 人，业务变化不大，事故频率低
  • 淘汰理由：故障频率到一定程度后修补速度赶不上规则腐化速度，复盘补的规则会重复覆盖既有体系，进一步推高重复率；75 天失效事件就是典型征兆

方案 B：全量迁到一套（如 Prometheus + Grafana OnCall）
#

把业务侧 SQL 轮询规则改写成 Prometheus rule，云厂商告警全部接 YACE 走 Prom 链路，最终统一收敛到 Grafana OnCall 做事件管理和排班。

  • 适用：有 1-2 人专职 SRE，半年到一年专项治理预算
  • 淘汰理由：业务侧 103 条规则中有 30 条以上是基于业务表的复合 SQL（如"某项目消息异常率 + 计费状态联合判断"），改写成 PromQL 需要先把业务指标 emit 成 metric，开发改造成本不可控；上线冻结期不能动业务代码

方案 C：保留两套但治理 + 互通（推荐）
#

承认两套体系各有合理性：Prometheus 擅长基础设施 + 时间序列指标，业务侧 SQL 轮询擅长复合业务条件 + 跨表 join。治理的目标是让两套各司其职、明确分工，并通过统一通知渠道收敛、统一 silence 入口、定期对账。

  • 适用：本文场景。已有两套并行体系，业务侧规则不易迁出
  • 核心动作：渠道收敛、Watchdog 心跳护栏、silence 规范、规则总线对账
  • 代价：需要持续维护对账文档，规则边界靠纪律而非工具强制

关键决策点：

  • Alertmanager 是唯一收敛点：不论告警从哪里产生，最终都进入 Alertmanager。业务侧告警通过 webhook 桥接（POST 到 Alertmanager API v2 /api/v2/alerts）注入。这样 silence、inhibit、route、模板渲染只有一处。
  • prometheus-webhook-dingtalk 是唯一钉钉出口：抛弃 prometheusalert（v4.9.1，2024-06 后无维护），统一用 timonwong/prometheus-webhook-dingtalk。
  • Watchdog 永远在线：一条 alert: Watchdog 始终 firing，配单独的 receiver 推 healthchecks.io。30s 收不到心跳，反向告警。
  • 新告警绑独立通道：任何 AWS / 阿里云资源告警，新建 SNS topic 和 Lambda，绝对不复用历史 Lambda。

实施步骤
#

1. 摸底：导出全部规则做分类
#

# us-prod 集群
kubectl -n monitoring get prometheusrule -o yaml > /tmp/rules-us-prod.yaml

# sandbox 集群
kubectl -n monitoring get prometheusrule -o yaml > /tmp/rules-sandbox.yaml

# 业务侧规则从 DB 直接 dump
psql -h <pg-host> -U <user> -d goalfy_monitor \
  -c "\copy (select id, name, severity, enabled, expression from monitor_rules) to '/tmp/rules-biz.csv' csv header"

按四个维度分类做表：

来源条数覆盖域重叠对象
Prom us-prod83基础设施 + 流量 + 中间件与业务侧的 5xx / 延迟规则重叠
Prom sandbox12沙箱容量 + 操作延迟无
业务侧 SQL103Agent / Gateway / 计费 / 沙箱业务延迟沙箱操作延迟与 Prom sandbox 重叠
CN CMS~20阿里云资源原生告警与 YACE 采集的资源告警重叠

明确划线：基础设施 + 时序 → Prom，业务复合 → 业务侧，云资源 → 统一走 YACE，三方各自不要跨界。

2. 渠道梳理：列出所有 SNS topic / Lambda / webhook
#

# AWS 侧
aws sns list-topics --region us-west-2 \
  | jq -r '.Topics[].TopicArn' | grep -i alert

aws sns list-subscriptions --region us-west-2 \
  | jq '.Subscriptions[] | select(.Endpoint | test("lambda")) | {topic: .TopicArn, lambda: .Endpoint}'

aws lambda list-functions --region us-west-2 \
  | jq -r '.Functions[].FunctionName' | grep -i alert

每个 Lambda 逐个看代码。重点找：

  • 硬编码的 title（如 "RabbitMQ 告警: ..."）
  • 硬编码的 broker / cluster / instance 字段
  • 硬编码的钉钉 webhook 或 token

发现这种"专用 Lambda"立即记入"禁止复用清单"。新建告警一律走独立 SNS + 独立 Lambda（或者更彻底的，全部走 YACE → Prom → Alertmanager 路径，绕开 Lambda 转发）。

3. silence 规则审计：找出隐藏的"误伤面"
#

# 列出当前所有 silence
amtool silence query --alertmanager.url=http://alertmanager:9093

# 重点找两类高危 matcher
amtool silence query --alertmanager.url=http://alertmanager:9093 \
  | grep -E 'alertname=~".+"|alertname=~".\*"'

任何用 .+ 或 .* 全匹配的 silence 都是潜在误伤。改写为负匹配（Alertmanager v2 API 支持 isEqual: false）：

{
  "matchers": [
    {"name": "alertname", "value": "Watchdog", "isRegex": false, "isEqual": false}
  ],
  "startsAt": "...",
  "endsAt": "...",
  "comment": "排除 Watchdog 心跳"
}

4. Watchdog 心跳验证告警链路存活
#

Alertmanager config 加一条恒定 firing 的告警和单独 receiver：

receivers:
- name: "dead-mans-switch"
  webhook_configs:
  - url: 'https://hc-ping.com/<uuid>'
    send_resolved: false
- name: "webhook-dingtalk-observe"
  webhook_configs:
  - url: 'http://prometheus-webhook-dingtalk:8060/dingtalk/observe/send'
    send_resolved: true
    max_alerts: 10

route:
  receiver: "webhook-dingtalk-observe"
  repeat_interval: 30m
  routes:
  - match:
      alertname: Watchdog
    receiver: "dead-mans-switch"
    group_wait: 10s
    group_interval: 1m
    repeat_interval: 1m

PrometheusRule 配套：

- alert: Watchdog
  expr: vector(1)
  labels:
    severity: none
  annotations:
    summary: "Watchdog heartbeat"

healthchecks.io 上配 1 分钟周期 + 2 分钟 grace，超时反向通知到独立的钉钉群（不能跟主告警同群，否则 Alertmanager 自己挂了通知不出去也无感）。

5. 渠道收敛：收口到 prometheus-webhook-dingtalk
#

新增钉钉 target 时按这个 config 模板：

targets:
  observe:
    url: <robot-webhook>
    secret: <signing-secret>
    message:
      title: '{{ template "default.title" . }}'
      text:  '{{ template "default.content" . }}'

钉钉机器人三个关键词必须同时配齐：告警 / 运维 / 恢复。理由见踩坑 6。

6. 规则归档与对账
#

把所有 PrometheusRule yaml 入 GitOps：

gitops-repo/
└── clusters/
    └── us-prod/
        └── monitoring/
            ├── p0-alerting-rules.yaml      # 12 条灾难检测
            ├── application-alerts.yaml     # 22 条 K8s 运行时
            ├── middleware-alerts.yaml      # 18 条中间件
            ├── aws-resources-alerting.yaml # 7 条 AWS 资源
            ├── blackbox-probes.yaml        # 5 条外部可达性
            └── ...

业务侧 103 条规则在 DB 里，导出 csv 入仓库做对账文档（不要求双向同步，只要求每周对账）。每条规则在仓库里维护四个字段：规则名 / 来源（Prom-us-prod / Prom-sandbox / 业务侧 / CN-CMS） / 责任人 / 处置文档链接。新人接手 on-call 第一件事就是读这份对账表，比读规则 yaml 高效得多。

7. 每周巡检 silenced + 失效规则
#

写一个简单脚本，每周一早晨跑，输出推到群里让团队 review：

# 长期 silence 提醒（>7 天的 active silence）
amtool silence query --alertmanager.url=http://alertmanager:9093 \
  --json | jq '.[] | select(.status.state=="active")
              | select((.endsAt | fromdateiso8601) - (.startsAt | fromdateiso8601) > 604800)'

# 永久 silence 检测（endsAt 超过当前时间一年以上的）
amtool silence query --alertmanager.url=http://alertmanager:9093 \
  --json | jq '.[] | select(.status.state=="active")
              | select((.endsAt | fromdateiso8601) - now > 31536000)'

# 失效规则检测（state=inactive 但创建超 30 天且从未 fire）
# 通过 Prometheus rules API + ALERTS_FOR_STATE metric 联合判断
curl -s http://prometheus:9090/api/v1/rules \
  | jq '.data.groups[].rules[] | select(.type=="alerting")
        | {name, state, lastEvaluation}'

# 规则总数对账（git 里的 yaml 数 vs 集群实际加载数）
git_rules=$(grep -rh '^- alert:' gitops-repo/clusters/us-prod/monitoring/ | wc -l)
api_rules=$(curl -s http://prometheus:9090/api/v1/rules \
            | jq '[.data.groups[].rules[] | select(.type=="alerting")] | length')
[ "$git_rules" -eq "$api_rules" ] || echo "DRIFT: git=$git_rules api=$api_rules"

巡检脚本本身也走 GitOps，CronJob 跑在监控集群里。一两周跑下来，75 天失效那种问题不会再发生。每月做一次更深入的回顾：把当月触发过的规则按"P0 命中数 / 误报数 / 静默次数"排序，找前三名做调优。

8. 老 CMS / Lambda 渠道的退役计划
#

历史 CloudWatch + Lambda 通道（如硬编码 RabbitMQ 字样的 Lambda）短期保持现状不动，避免影响在用告警，但新规必须严格执行：

  • 任何新增 AWS 资源告警一律走 YACE → Prom → Alertmanager 路径
  • 必须新建 CloudWatch alarm 时，新建独立 SNS topic（如 ops-aws-alerts）+ 独立 namespace-aware Lambda
  • Lambda 模板里禁止硬编码业务名 / 集群名 / Broker 名，所有字段从 SNS 消息体解析

冻结期过后，把残留 3 条 RabbitMQ CloudWatch alarm 一并迁到 YACE，旧 Lambda 退役。退役前先在新通道并行运行 2 周，对账无差异再下线。

踩过的坑
#

坑 1：prometheusalert URL encode 把钉钉 token 吞了
#

现象：钉钉返回 {"errcode":300005,"errmsg":"token is not exist"}，token 直接 curl 测有效。

根因：Alertmanager webhook URL 嵌套了钉钉 URL：

http://prometheusalert:8080/prometheusalert?type=dd&tpl=prom-dd&ddurl=https://oapi.dingtalk.com/robot/send?access_token=XXXX

prometheusalert 的 query parser 遇到内层 ?access_token=... 当作第二个 query 起始，ddurl 的 value 只取到 https://oapi.dingtalk.com/robot/send，token 丢了。

修复：URL encode ddurl 值。但更彻底的方案是干脆换 adapter：换成 timonwong/prometheus-webhook-dingtalk，直接在 config 里定义 target，不存在 URL 嵌套。

通用结论：钉钉报 300005 但 token 单独可用，第一反应查 URL encode；选告警 adapter 时优先选 config-as-data 而不是 config-as-URL 的方案。

坑 2：silence 用 .+ 误伤 Watchdog 心跳
#

现象：上线日临时 silence 所有 active 告警，写了 alertname=~".+"，结果反向告警机制失效几小时没人发现，期间 Alertmanager 进程被人意外 kill 也无感。

根因：.+ 匹配所有 alertname，包括恒定 firing 的 Watchdog。Watchdog 被 silence → healthchecks.io 收不到心跳 → 但反向告警的群跟主告警同群，主链路已经 silence 了什么也收不到。

修复：所有"全部 silence"动作必须排除 Watchdog，用负匹配：

amtool silence add --alertmanager.url=http://alertmanager:9093 \
  alertname!=Watchdog \
  --comment="上线窗口 silence" \
  --duration=2h

通用结论：任何 silence “除了 Watchdog 之外” 必须显式写 alertname!=Watchdog；反向告警通道必须独立于主告警通道（不同群、不同 webhook）。

坑 3：Aurora DeletionProtection 是 cluster 级属性
#

现象：执行 aws rds modify-db-instance --db-instance-identifier <aurora-instance> --deletion-protection 后无报错，但 protection 状态没变。

根因：Aurora 的 instance 是 cluster 的 member，deletion-protection 这类属性挂在 cluster 层，不在 instance 层。命令调用没报错是因为字段允许，但实际写入到 instance 元数据里被忽略。

修复：

# Aurora 改 cluster 层
aws rds modify-db-cluster \
  --db-cluster-identifier <cluster-id> \
  --deletion-protection

# 非 Aurora（如 PG 单实例）才改 instance 层
aws rds modify-db-instance \
  --db-instance-identifier <instance-id> \
  --deletion-protection

通用结论：动 Aurora 任何属性前先确认是 cluster 级还是 instance 级；不要把单实例 RDS 命令套到 Aurora 上。延伸到告警，YACE 采集 Aurora 指标时也要分清 aws_rds_* 和 aws_rds_cluster_* 是不同的 metric 命名空间。

坑 4：ApplicationSet “复活” 删除的告警 App
#

现象：删一个老的 alert-related ArgoCD Application，argocd app delete 或 kubectl delete application 都执行成功了，但几秒后 app 又出现，状态 deletionTimestamp=null。

根因：这个 Application 的 ownerReferences 指向某个 ApplicationSet，ApplicationSet controller 检测到 child app 缺失立即重建。kubectl delete 改不了源头。

修复：改 git 源头。要么删 clusters/<env>/applications/<path>/ 目录让 ApplicationSet generator 不再产出这个 app，要么改 generator 的过滤规则。流程是：

# 先确认是不是 ApplicationSet 生成的
kubectl get application <name> -n argocd -o yaml | grep -A 3 ownerReferences

# 如果有 ownerReferences 指向 ApplicationSet，改 git 不要 kubectl

通用结论：删任何 ArgoCD Application 之前先 check ownerReferences；GitOps 体系下 kubectl delete 大概率被 controller 回滚，源头永远在 git。

坑 5：钉钉关键词只覆盖 firing「告警」，resolved「恢复」全被拒
#

现象：webhook-dingtalk 日志里大量 respCode=310000 关键词不匹配，每 5 分钟（group_interval）一波。但表面看 firing 告警能正常推到群。

根因：模板里 firing 渲染成 ## 【告警】xxx（含"告警"，过关），resolved 渲染成 ## 【恢复】xxx（只有"恢复"，不含"告警/运维"，被拒）。钉钉机器人原配的关键词只有 告警 和 运维，send_resolved: true 所以每次 alert 转 resolved 都触发恢复推送，全被钉钉拒。业务方以为告警没了，实际从来没收到恢复消息，这是真正的运维黑洞。

钉钉关键词检测的是 markdown.text 字段内容（不是 markdown.title），title 写得再对也没用。

修复：给钉钉机器人加关键词 恢复，三个关键词 告警/运维/恢复 全覆盖。或者改模板把 【恢复】 替换成 【告警恢复】，既覆盖关键词又表意清晰。

通用结论：新建钉钉机器人时关键词必须同时覆盖 firing 和 resolved 两类消息体里出现的所有标题词；推 webhook-dingtalk 部署 / 切群时必须 curl 测两类消息（firing 模板 + resolved 模板）的 happy path，不能只测 firing；看到 webhook-dingtalk 日志稳定每 group_interval 一波 310000，第一反应是 resolved 通知被拒。

坑 6：老 Lambda 模板硬编码业务名，新告警绑上去全错位
#

现象：新增 Aurora CPU 告警走 SNS topic rabbitmq-prod-alerts 复用 Lambda rabbitmq-dingtalk-alert，钉钉收到的消息是：

【告警】RabbitMQ 告警: aurora-cpu-high
Broker: prod-xxx-rabbitmq-ha-v2 (m5.large)
Metric: CPUUtilization

业务跟字段全乱。

根因：Lambda 代码里硬编码了 title = "RabbitMQ 告警: ..." 和 broker / instance 字段，任何非 RabbitMQ 告警走这个 Lambda 都会被渲染成误导消息。

修复：

  • 短期：新增 AWS 资源告警一律走 YACE → Prometheus → Alertmanager → webhook-dingtalk 路径，绕开 Lambda
  • 必须新建 CloudWatch alarm 时，新建独立 SNS topic（如 ops-aws-alerts）+ 独立 namespace-aware Lambda，绝对不往 rabbitmq-prod-alerts 挂
  • 老的 3 条 RabbitMQ 告警保持现状别动，等下次治理窗口连 Lambda 一并退役

通用结论：任何带"业务名"的资源（Lambda / SNS topic / IAM role / Secret）都默认有硬编码风险，复用前先看代码；告警渠道命名要 namespace-aware（ops-<service>-alerts），不要按"恰好已经有了"复用。

坑 7：Prometheus Operator configmap 挂载热更新延迟
#

现象：改完 webhook-dingtalk 的 config / template configmap，POST /-/reload 后还是用老模板，钉钉群里收到的格式没变。

根因：

  • kubelet configmap sync period 默认 ~60s，pod 内 mounted 文件还是旧的
  • webhook-dingtalk 的 /-/reload 只重加载 config.yml，不含 templates 目录

修复：改完 configmap 直接 kubectl rollout restart deploy webhook-dingtalk，让 pod 重启时重新挂载 + 完整加载所有模板文件。

通用结论：configmap 内容是模板/脚本时优先 rollout restart 而非 reload。reload 只对 config 文件本身生效，不对 mounted 目录里的辅助文件生效。

坑 8：sandbox 告警规则因 label 不匹配 75 天没生效
#

现象：sandbox 集群有 12 条 PrometheusRule yaml 在 git 里好好放着，集群里也都创建出来了，但日常运维里从来没有触发过任何一条。一次专项巡检才发现：75 天里这些规则一次都没被评估过。

根因：Prometheus Operator 通过 ruleSelector 选取 PrometheusRule，selector 是 app=monitoring，但这批 yaml 的 metadata.labels 写的是 release=monitoring，label 不匹配，Operator 直接跳过这批规则。规则在 kubectl get prometheusrule 里能看到，但没注入 Prometheus 配置。

修复：

# 先确认 selector
kubectl -n monitoring get prometheus -o jsonpath='{.items[0].spec.ruleSelector}'

# 修 yaml label 匹配 selector，重新 apply
# 同时验证 Prometheus 是否实际加载
curl -s http://prometheus:9090/api/v1/rules | jq '.data.groups[].name'

通用结论：写完 PrometheusRule 后必须从 Prometheus /api/v1/rules 接口验证规则真的被加载，不能只看 kubectl get prometheusrule；每周巡检脚本要加一条"规则总数 vs 集群规则总数对账"。

  • 新人入职第一周就能独立处理告警，不需要找老员工问"这条 RabbitMQ 告警是不是新加的 Aurora 告警"

局限
#

下面这些场景本方案不直接适用：

  • 真正多 region active-active 部署：本方案隐含一个"主集群 + 沙箱集群 + 云监控"的中心化模型。多 region 双活场景需要每个 region 一套 Alertmanager + 跨 region cluster mode，复杂度上一个量级
  • 替代真正的可观测性平台：本方案处理的是告警治理，不替代 Datadog / New Relic / Grafana Cloud 这类一站式可观测性 SaaS。如果团队需要 trace + log + metric 一体化分析、AIOps 异常检测、自动根因分析，本方案做不到
  • 超大规模（>1000 条规则 / >50 个集群）：当规则规模到这个量级，纯靠 GitOps + 文档对账已经维护不动，需要引入告警 as code 框架（grizzly / cortex-ruler tooling）和告警生命周期工具
  • 强合规审计场景：本方案的 silence 审计依赖巡检脚本，不带审批流和自动过期。金融/医疗等强合规场景需要专门的告警工单系统

后续演进方向
#

接下来 6-12 个月可以做的事：

  1. on-call rotation 自动化：当前 7×24 单人随时响应不可持续。集成 Grafana OnCall 或 PagerDuty 做排班 + 升级路径，告警按严重度走不同人
  2. 告警分级 SLO 化：当前 P0/P1/P2 是按经验切的，可以基于 SLI（如月可用性 99.9%）+ error budget 做严格分级，error budget 烧得快才报 P0
  3. silence 强制 endsAt + 工单化：下一版的 silence 操作只能通过工单走（写明 owner / 影响面 / 自动过期时间），不允许 amtool 直接下手
  4. AI 摘要告警风暴：钉钉群里短时间一波告警时，调 LLM 自动生成"这一波告警在描述同一个故障"的归因摘要，减轻 on-call 阅读负担
  5. 业务侧规则的 promQL 化：上线冻结期过后，挑选业务侧 103 条里耦合度最低的 30 条改写成 PromQL（要求开发先 emit 业务 metric），逐步收缩业务侧规则到只剩跨表 join 的部分
  6. 跨云告警延迟 SLO：CN 云监控 → 钉钉的链路延迟没人测过，可以加一条端到端延迟 metric（注入测试告警 → 钉钉群消息时间戳）做 SLO

最后验证：2026-04-30，Prometheus 2.54 / Alertmanager 0.27 / prometheus-webhook-dingtalk 2.1.0 / amtool 0.27。超过 12 个月后慎重参考。
