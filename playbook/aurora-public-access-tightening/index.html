元信息

  • 适用规模：10-200 人团队、单 Region 或多 Region AWS 部署
  • 适用云：AWS（阿里云 RDS 思路相似但 SG 模型不同）
  • 运维负担：实施期 3-5 个工作日，长期维护极轻
  • 月成本：增量约 $0（VPC Peering 跨 AZ 流量、SSM 跳板机微量开销）
  • 最后验证：2026-04-30，AWS CLI 2.17 + Aurora MySQL 8.0

适用场景
#

满足以下任意两条，建议按本 Playbook 推进：

  • Aurora/RDS Security Group 存在 0.0.0.0/0 全协议或全 3306 端口规则
  • SG 上存在来源不明的 IP /32 白名单（多年累积、无人维护）
  • 跨 Region 业务通过公网 hostname 直连数据库
  • 团队既要关公网入口又要保留开发者本地调试能力
  • 已经吃过一次安全审计的批评，但担心动手会打断业务

不适用场景见文末「局限」一节。

核心问题
#

公网暴露的 Aurora 在生产环境实际形态通常是「三层叠加」，比单纯一条 0.0.0.0/0 更难处理：

  1. 全协议公网开放：SG 入站存在 protocol -1, source 0.0.0.0/0，意味着不只是 3306，而是所有端口对任意来源放开。这种规则常常是早期为方便测试加的，没人敢删。Shodan、Censys 这类公网扫描器会把它收录，定期被探针扫到留下 RDS 实例的版本指纹和元数据。
  2. 来源不明的 IP 白名单：常见一两条 47.x.x.x/32 或 139.x.x.x/32 这样的规则，可能是某次本地调试加的，可能是已经离职同事的家庭宽带 IP，也可能是某家第三方服务（实际遇到过 Linode 主机房 IP 段，初步判断是历史合作方的爬虫机器，但已无法考证）。这类规则在公网开放期间被淹没在背景流量里，关公网后才会显现。
  3. 跨 Region 业务依赖公网 hostname：例如 ap-southeast-1 的某服务在 Nacos 配置里直接用 prod-aurora.example.aws.com 这个公网域名连 us-west-2 的主库。一旦关公网，这条链路立刻断。这种依赖往往在迁移决策做出前没人记得，Nacos 配置中心里 grep 一遍才会浮出来。

为什么这个状态长期存在？因为单看任何一条规则，删掉都"可能影响某个东西"，于是大家都不动。久而久之责任分散，最终没人能回答"我可以删 0.0.0.0/0 吗"这种问题。突破口是用证据替代猜测——把"我不知道有谁在用"变成"Flow Logs 三周内来自公网的入流量是哪几个 IP，每个 IP 我能不能找到归属"。

期望状态：

  • SG 入站只允许同 Region VPC CIDR 和已建立 Peering 的 VPC CIDR
  • 跨 Region 业务走 VPC Peering + Private Hosted Zone（PHZ），从配置上根本无法触达公网入口
  • 开发者本地工具仍能在受控前提下访问 prod 数据库
  • 所有访问可审计、权限可回收
  • 收紧动作可以在不出现明显业务停机的前提下推进

方案对比
#

讨论过三个方向，最终选定第三个并预留向更彻底方案演进的空间。

方案 A：维持公网 + 收紧 IP 白名单
#

保留 Aurora 公网，把 0.0.0.0/0 替换为办公网络 CIDR、VPN 出口 IP、若干开发者家宽 IP。

  • 适用：团队 5 人以下、办公地点固定、无远程办公
  • 淘汰理由：远程办公 + 居家办公 IP 漂移频繁，白名单维护工作量随团队增长爆炸；Aurora 公网入口本身仍是攻击面，被 Shodan、Censys 持续扫描；不解决跨 Region 服务依赖公网 hostname 的问题

方案 B：架设 VPN 集中接入
#

部署 AWS Client VPN 或 OpenVPN/WireGuard，所有开发者先连 VPN 再访问数据库。

  • 适用：单 Region、单云、团队稳定的中型公司
  • 淘汰理由：AWS Client VPN 按连接小时收费，30 人月成本就到 ¥12k 量级；自建 VPN 单点故障重；多 Region 多云场景下 VPN 网关数量翻倍；新人接入仍需配 VPN client + 证书 + 路由

方案 C：渐进式收紧 + 零信任 mesh（选定）
#

分阶段推进：

  1. 先把跨 Region 服务从公网 hostname 切到 VPC Peering + 私有 DNS
  2. 给开发者准备 SSM Port Forwarding 作为公网替代
  3. 原子切换 SG 规则（先加白后删开放）
  4. 清理长尾 IP 白名单
  5. 中长期推 Headscale/Tailscale mesh，把 SSM 也淘汰掉

这个路径的好处：每一步都可独立验证、可独立回滚；前四步全是 AWS 原生能力，不引入新依赖；mesh 单独是另一个 Playbook（见文末「后续演进」），收紧本身不依赖 mesh 落地。

整个推进周期短则 3-5 个工作日（如果跨 Region 依赖少、开发者群体熟悉 AWS CLI），长则 2-3 周（涉及多个团队协调、多个跨 Region 链路梳理）。本 Playbook 给出的是技术路径，组织协调时间需要按团队规模额外预留。

推荐架构
#

收紧过程中存在三个有意义的阶段，每个阶段都能稳定运行。

flowchart LR
    subgraph S0["阶段 0：初始公网状态"]
        DEV0[开发者笔记本] -->|公网直连 3306| AUR0[(Aurora<br/>0.0.0.0/0 开放)]
        XR0[跨 Region 服务] -->|公网 hostname| AUR0
        BAD[Shodan/扫描器] -.探测.-> AUR0
    end
    subgraph S1["阶段 1：跨 Region 切私网 + SSM 替代"]
        DEV1[开发者] -->|SSM Tunnel| BASTION[SSM 跳板 Pod/EC2]
        BASTION -->|VPC 内| AUR1[(Aurora<br/>VPC CIDR + 旧白名单)]
        XR1[跨 Region 服务] -->|VPC Peering<br/>Route53 PHZ| AUR1
    end
    subgraph S2["阶段 2：SG 收敛 + 长尾清理"]
        DEV2[开发者] -->|SSM Tunnel| BAST2[SSM 跳板]
        BAST2 -->|VPC 内| AUR2[(Aurora<br/>仅 VPC CIDR)]
        XR2[跨 Region 服务] -->|VPC Peering| AUR2
    end
    S0 ==>|跨 Region 切私网<br/>SSM 上线| S1
    S1 ==>|删 0.0.0.0/0<br/>清理长尾白名单| S2

关键决策点：

  • 阶段 1 是稳定可运行的过渡态，可以在这里停留几周观察业务表现，不必赶着进阶段 2
  • 跨 Region 服务的切换必须先做，因为它对业务有真实影响；SG 规则收敛是后置动作
  • SSM 通道能力先到位，避免删 SG 之后才发现开发者本地工具全部失联

数据流要点：

  • 同 Region 业务服务一直走 VPC 内私网，本来就不依赖公网入口，整个收紧过程对它们透明
  • 跨 Region 业务先建 Peering + PHZ，再改 Nacos/配置中心里的 host，确认日志中显示新 host 解析成功后才动 SG
  • 开发者工具链改用 SSM 后，AWS IAM 取代 SG IP 白名单成为权限边界

实施步骤
#

第 1 步：摸底 — 谁在用公网入口
#

收紧前必须用证据替代猜测，三个数据源足够：

VPC Flow Logs 看入向源 IP 分布

# 假设 Flow Logs 已开启并存到 S3
aws logs start-query \
  --log-group-name /aws/vpc-flow/prod \
  --start-time $(date -d '7 days ago' +%s) \
  --end-time $(date +%s) \
  --query-string 'fields srcAddr, dstPort, action
                  | filter dstAddr = "10.x.y.z" and dstPort in [3306, 5432]
                  | stats count() by srcAddr, action
                  | sort count desc'

输出常见三类来源：内部 VPC IP（合规、不动）、Peering VPC IP（合规、不动）、公网 IP（重点关注）。把公网 IP 反查 PTR 和 ASN：内部 VPN 出口（OK）、办公网络（OK）、不认识的 IP（高优先级排查）。

SG 历史规则审计

aws ec2 describe-security-groups --group-ids sg-xxxxxxxxxxxxx \
  --region us-west-2 \
  --query 'SecurityGroups[0].IpPermissions' --output json

CloudTrail 里 AuthorizeSecurityGroupIngress 事件能查到每条规则的添加时间和操作人。describe-security-group-rules 能看到 rule 创建时的 description（如果之前的同事有写就万幸）。

应用配置反查

grep 所有 Nacos / Apollo / 环境变量里的数据库 host，找出所有依赖 *.rds.amazonaws.com 公网域名的服务。

# 在配置中心导出当前命名空间所有配置后批量 grep
grep -rE 'rds\.amazonaws\.com|aurora.*\.cluster' ./nacos-export/ \
  | awk -F: '{print $1}' | sort -u

输出的文件列表就是必须改 host 的"作业清单"。这一步不能跳——SG 切了之后才发现某个偏门服务还连公网域名，会变成线上事故。

第 2 步：跨 Region 服务切 VPC Peering + Route53 PHZ
#

这是收紧动作中风险最高的一步，必须先做并稳定运行 1-2 天。

建立 Peering：us-west-2 prod VPC（10.3.0.0/18）和 ap-southeast-1 pre VPC（10.52.0.0/16）。两侧路由表添加对端 CIDR 路由。

建 Private Hosted Zone：

# 在 ap-southeast-1 创建 PHZ
aws route53 create-hosted-zone \
  --name rds.internal --hosted-zone-config PrivateZone=true \
  --vpc VPCRegion=ap-southeast-1,VPCId=vpc-xxx \
  --caller-reference $(date +%s)

# 加 A 记录指向 Aurora writer 的私有 IP
aws route53 change-resource-record-sets \
  --hosted-zone-id Z0XXXXXXXXXXXX \
  --change-batch '{
    "Changes": [{
      "Action": "CREATE",
      "ResourceRecordSet": {
        "Name": "aurora-prod.rds.internal",
        "Type": "A", "TTL": 60,
        "ResourceRecords": [{"Value": "10.x.y.z"}]
      }
    }]
  }'

重要：Aurora cluster endpoint 在故障切换时会指向新的 writer，私有 IP 会变。生产场景两个做法：

  • 短期：用 cluster endpoint 的私有 IP，加监控，故障切换时人工更新 PHZ 记录
  • 长期：在 VPC 里跑一个 unbound/CoreDNS 转发 *.rds.amazonaws.com 解析到 VPC resolver，用真实的 cluster endpoint hostname 即可

本次落地时业务 Aurora 主备故障切换频率极低，先采用短期方案 + 告警监控。

Aurora SG 加 Peering 端白名单：

aws ec2 authorize-security-group-ingress \
  --group-id sg-xxxxxxxxxxxxx \
  --protocol tcp --port 3306 --cidr 10.52.0.0/16 \
  --region us-west-2

改业务配置：把 Nacos 里这条服务的 dataId 中 mysql.host 从公网域名改成 aurora-prod.rds.internal，rollout restart。看 Pod 启动日志确认 MySQL connected to host: aurora-prod.rds.internal。

观察 1-2 天，确认无连接抖动后进入下一步。具体观察哪些指标：

  • Aurora DatabaseConnections 指标曲线无异常下降
  • Aurora Aborted_connections / Aborted_clients 状态变量无突增
  • 跨 Region 服务的 Pod 日志中持续出现新 host 解析成功记录
  • 应用层错误率（5xx）无明显波动

这一步切完，公网 hostname 仍然可达（SG 还没改），意味着如果有遗漏的跨 Region 服务，它仍然能正常工作，给了一个隐性的兜底窗口。

第 3 步：开发者 SSM Port Forwarding 替代方案
#

目标是让"原本在笔记本上 mysql -h prod-aurora.example.aws.com“的工作流，变成”mysql -h 127.0.0.1 -P 13306"，背后透明地走 SSM 隧道。

准备跳板：选一台 prod VPC 内的 EC2 节点（或新建 t4g.nano）作为跳板，确认它的 SSM Agent 是 Online。给该实例的 IAM Role 加 AmazonSSMManagedInstanceCore。

封装 db-tunnel.sh：

#!/bin/bash
# db-tunnel.sh prod | qa | pre
set -euo pipefail
ENV=${1:-prod}
case "$ENV" in
  prod) BASTION="i-xxxxxxxxxxxxx"; TARGET_HOST="aurora-prod.rds.internal"; LOCAL_PORT=13306 ;;
  qa)   BASTION="i-yyyyyyyyyyyyy"; TARGET_HOST="qa-aurora.rds.internal";   LOCAL_PORT=13307 ;;
  *) echo "unknown env"; exit 1 ;;
esac
aws ssm start-session \
  --target "$BASTION" \
  --document-name AWS-StartPortForwardingSessionToRemoteHost \
  --parameters "host=${TARGET_HOST},portNumber=3306,localPortNumber=${LOCAL_PORT}" \
  --region us-west-2

IAM Group：建一个 db-access group，绑一个最小权限 policy db-tunnel-ssm-access，只允许对指定跳板实例 ID 调用 ssm:StartSession + ssm:TerminateSession，不给任何 RDS 权限。新人接入只要 aws iam add-user-to-group --group-name db-access --user-name xxx 一行。

分发：脚本放私有 S3 bucket，预签名 URL 24 小时有效。或者塞进内部 Wiki。

给开发者一段 cheat sheet：

# 一次性配置
curl -o ~/db-tunnel.sh <预签名 URL> && chmod +x ~/db-tunnel.sh

# 使用
~/db-tunnel.sh prod &     # 后台启动隧道
mysql -h 127.0.0.1 -P 13306 -u <readonly_user> -p

让 5-10 个高频使用者先试，覆盖 DataGrip / DBeaver / 命令行 / 应用本地启动等场景，确认所有工作流可行后再推 SG 收敛。

注意会失败的两类工具：少数 GUI 客户端默认不允许"127.0.0.1 + 非标准端口 + SSL 必须"组合，需要在连接配置里手动关闭 SSL 校验或导入 RDS CA 证书；某些 ORM 框架会把 host 解析结果缓存，本地端口转发后第一次连不上需要重启进程。这些坑都很短小，但放在一起会让"开发者第一次跑 SSM 隧道"的体验变差。提前在团队 wiki 集中记录，避免每个人都踩一遍。

第 4 步：SG 规则原子切换（先加白后删开放）
#

按"先加再删"顺序，避免任意时刻的连接黑洞。

# 4.1 加白同 VPC + Peering 网段
for cidr in 10.3.0.0/18 10.2.0.0/18 10.52.0.0/16; do
  aws ec2 authorize-security-group-ingress \
    --group-id sg-xxxxxxxxxxxxx \
    --protocol tcp --port 3306 --cidr $cidr \
    --region us-west-2
done

# 4.2 验证：从同 VPC 节点 / Peering 对端 / SSM 跳板各连一次
mysql -h aurora-prod.rds.internal -u xxx -p -e "SELECT 1"

# 4.3 删 0.0.0.0/0 全协议
aws ec2 revoke-security-group-ingress \
  --group-id sg-xxxxxxxxxxxxx \
  --protocol -1 --cidr 0.0.0.0/0 \
  --region us-west-2

# 4.4 立刻验证业务无异常
# - 看 Aurora "Database Connections" CloudWatch 指标，曲线应该没波动
# - tail 应用日志，没有突增的 "connection refused" / "timeout"

切换窗口建议放业务低峰，但实际操作如果第 2 步、第 3 步都做扎实，业务感知接近零。

第 5 步：长尾 IP 白名单清理
#

0.0.0.0/0 没了，但通常 SG 上还残留几条 /32。逐一处理：

# 列出当前所有规则
aws ec2 describe-security-group-rules \
  --filters Name=group-id,Values=sg-xxxxxxxxxxxxx \
  --region us-west-2 \
  --query 'SecurityGroupRules[?IsEgress==`false`].[CidrIpv4, FromPort, ToPort, Description]' \
  --output table

对每条 /32 走流程：

  1. WHOIS / IP 反查 ASN，初步判断归属（运营商 / 云厂商 / 数据中心）
  2. 在团队 IM 里贴出来询问 24-48 小时
  3. 无人认领则改备注 pending-delete-2026-04-30，先保留 1 周观察 Flow Logs 是否还有该 IP 流量
  4. 一周后确认无流量，删除

实际操作中常遇到 Linode、DigitalOcean、阿里云国际等 IDC 的 IP，几乎都是历史合作方或外包团队遗留，无人能说出来源。这种规则只能按上述流程清理，没有捷径。

第 6 步：同步收紧 PostgreSQL（很多人会漏）
#

prod 环境通常不只 Aurora MySQL，还有 RDS PostgreSQL（业务 ai-gateway / agent 用），它的 SG 是独立的另一个：

# 找出 prod 内所有 RDS 实例的 SG
aws rds describe-db-instances \
  --region us-west-2 \
  --query 'DBInstances[].[DBInstanceIdentifier, VpcSecurityGroups[].VpcSecurityGroupId]' \
  --output table

对每个 SG 重复第 4-5 步。这一步在实际项目里被遗漏过——MySQL SG 处理完发周报，第二天才发现 PG SG 还挂着 0.0.0.0/0。

踩过的坑
#

坑 1：直接删 0.0.0.0/0 后跨 Region 服务断链
#

现象：在没做第 2 步的情况下直接删了 Aurora SG 的 0.0.0.0/0 规则。10 分钟内，部署在 ap-southeast-1 的某后台服务 service-foo 开始报 dial tcp: lookup prod-aurora.example.aws.com: i/o timeout。该服务依赖 prod 主库做用户中心数据同步，业务功能异常 30 分钟。

根因：服务的配置中心 host 是 Aurora 公网域名。SG 关公网后，该域名的公网 IP 仍能解析，但 TCP 三次握手在 SG 这一层被拒。跨 Region 服务没有 Peering 入口，于是完全失联。

修复：紧急回滚（重新加 0.0.0.0/0），按本 Playbook 第 2 步建立 Peering + PHZ，改 Nacos host 为 aurora-prod.rds.internal，确认日志后再次执行第 4 步。

通用结论：SG 收敛永远是最后一步。任何跨 Region / 跨账号的服务依赖都必须先迁移到私网通信，迁移完确认稳定才能动 SG。回滚要预先准备好脚本，能在 60 秒内重加规则。

坑 2：Linode IP 白名单的归属之谜
#

现象：SG 上有一条 139.x.x.x/32 → 3306 规则，CloudTrail 看添加于两年前，操作账号是已离职同事。WHOIS 显示属于 Linode 某数据中心。团队 IM 询问一周无人认领，Flow Logs 显示该 IP 月均确实有几千条到 3306 的连接。

根因：进一步排查发现是当年某个外包做爬虫的合作方，连这家合作方现在还在不在合作都查不清楚。这条白名单本质上是一个"活着的不明流量入口"，在公网开放期间被淹没在背景噪音里没人注意，公网关闭后才显现。

修复：联系业务负责人确认外包合作方早已结束，删除规则。删除后业务无任何反馈。

通用结论：SG 上每条 IP 白名单都应该有 description + ticket 引用 + 责任人。建议团队约定：新加规则时 description 必填，格式如 <责任人>-<工单号>-<到期时间>，到期前若需续期主动延长。配套定期审计脚本扫到期规则。

坑 3：SG 收紧后忘了 PostgreSQL
#

现象：MySQL SG 处理完，第二天审计发现 prod RDS PostgreSQL 还挂着 0.0.0.0/0 → 5432。这是一个独立 SG，跟 Aurora MySQL 完全无关，但服务于同一个 prod 业务集群。

根因：心智模型里把"prod 数据库"等同于"Aurora"，忽略了同环境下还有 PostgreSQL 实例。

修复：把 RDS PostgreSQL SG 也按第 4-5 步收敛。同时用脚本扫描所有环境的所有 RDS 实例 SG，输出还含 0.0.0.0/0 入站规则的清单。

aws rds describe-db-instances --region us-west-2 \
  --query 'DBInstances[].VpcSecurityGroups[].VpcSecurityGroupId' \
  --output text | tr '\t' '\n' | sort -u | while read sg; do
    open=$(aws ec2 describe-security-groups --group-ids $sg --region us-west-2 \
      --query 'SecurityGroups[0].IpPermissions[?contains(IpRanges[].CidrIp, `0.0.0.0/0`)]' \
      --output json)
    [ "$open" != "[]" ] && echo "$sg has 0.0.0.0/0"
done

通用结论：收紧动作要按"环境 × 数据库类型"二维矩阵覆盖，不要按记忆里的服务清单覆盖。CloudWatch 资源清单或 Steampipe 这类工具能快速生成全量清单。

坑 4：开发者本地连不上后的协调成本
#

现象：SG 收敛当天，多名开发陆续在 IM 里反馈"我连不上 prod 数据库了"，运维需要逐个回答"装 SSM Plugin、配 AWS Profile、跑 db-tunnel.sh"。最忙的两小时收到 8 个相似工单。

根因：第 3 步的"准备 SSM 替代方案"做得不够前置，公告发了，但只有少数高频用户提前迁移，多数人是 SG 关掉之后才被迫迁。

修复：

  • 在 SG 收敛前一周，IM 公告 + 强制要求所有 db-access 组成员提交一次"我已成功用 db-tunnel 连接 prod"截图
  • 配套 onboarding 脚本 setup-db-access.sh，自动检查 AWS CLI、Session Manager Plugin、IAM 凭据、跳板可达性，给出一键修复建议
  • 安排 30 分钟内部线上培训 + 录屏存档

通用结论：安全收紧的协调成本经常被低估。技术动作 1 小时完成，配套沟通要前置一周。给开发者准备好「替代工作流的完整路径」（不只是脚本，包括所有环境变量、IAM 配置、常见报错），比 SG 切换本身更重要。

衡量指标
#

收紧前后做 before/after 对比，下面是某个生产 Aurora SG 的真实数据（已脱敏）。

维度BeforeAfter
SG 入站规则数11 条（含 1 条 0.0.0.0/0 全协议）4 条（仅 VPC CIDR + 1 条临时 IP 待清）
公网入口数全协议（65535 端口）0
暴露端口1 - 65535仅 3306（且仅对内网 CIDR）
Shodan 扫描可见性持续被记录公网无响应，30 天内 Shodan 条目过期
跨 Region 调用方式公网 hostnameVPC Peering + Route53 PHZ
开发者直连方式IP 白名单 + 公网SSM Port Forwarding
权限边界SG IP 白名单（无身份）IAM Group + Session Manager（带身份）
审计能力仅 Aurora general log（默认不开）CloudTrail StartSession 事件 + IAM CloudTrail

定性变化：

  • 攻击面从 65535 端口对全网降到 0。后续若想进一步上 PrivateLink 把内部 traffic 也包起来，门槛已经很低
  • 权限审批从「网络位置」转为「身份」。同事离职、外包合作结束、临时调试需求等都能通过 IAM 在分钟级处理，不再需要去 SG 里找 IP
  • 跨云、跨 Region 一致。下一步给阿里云 RDS 做同样收紧时，思路完全可复用，只需替换 SG 模型为阿里云白名单组

局限
#

本 Playbook 不解决以下问题，请有针对性选择其他方案：

  • 跨境延迟：us-west-2 ↔ ap-southeast-1 的物理延迟 175ms 量级，VPC Peering 没法变魔术。如果业务对跨境延迟敏感，要从架构上做分库 / 数据下沉，而非纠结公网/私网
  • 纯 AWS 方案：阿里云 RDS、GCP CloudSQL、Azure Database 的安全模型不同（白名单组 / Authorized Networks / Firewall Rules），思路相同但命令完全不同
  • SSM 方案需要 K8s 节点或 EC2 跳板：完全 Serverless（Fargate-only）的环境需要专门部署一个跳板，或用 PrivateLink + Bastion Service
  • 本方案的终点不是零信任：SSM Port Forwarding 仍依赖 AWS IAM + SSM 这套，跨云不通用，新人接入步骤不算少。彻底零信任要走 mesh 方案
  • 不覆盖应用层 SQL 注入、慢查询攻击等问题：网络层收紧只是"让外人摸不到门"，应用层攻击需要 WAF / SQL 审计 / 慢查询防御等独立手段
  • PubliclyAccessible=false 是更彻底的方案：本 Playbook 只关 SG 入站，Aurora 实例本身的 PubliclyAccessible 属性默认为 true。下一步可以评估关闭它，彻底从 RDS 控制面层面消除公网入口

后续演进方向
#

短期 6-12 周：

  • 把 RDS / Aurora 实例的 PubliclyAccessible 设为 false，彻底关闭公网 endpoint
  • 推 IAM database authentication，逐步替代静态密码
  • 对 SG 规则启用 CloudWatch + Lambda 自动审计，新增 0.0.0.0/0 入站规则触发钉钉告警

中期 3-6 个月：

  • 上零信任 mesh（Headscale + Tailscale 或 Tailscale SaaS）。SSM 隧道作为 fallback 保留 1-2 个月后下线，开发者接入统一为「连 mesh → 直连 RDS 内网 IP」。本站另有一篇专门 Playbook 记录此方案的落地（标签 Headscale / 零信任）。
  • PrivateLink for RDS：跨账号场景下用 PrivateLink 提供更细粒度的服务级访问控制
  • 每月 SG 审计自动化：脚本扫所有 RDS / Aurora SG，发现 0.0.0.0/0 或长期未访问的 /32 自动出工单

长期：

  • 内部数据访问统一走身份 + 策略层（OPA/Cedar），SG 只剩兜底
  • 数据库连接审计接入 SIEM，离职、异常访问、批量导出等行为可秒级告警

最后验证：2026-04-30，AWS CLI 2.17、Aurora MySQL 8.0.mysql_aurora.3.08.0、Session Manager Plugin 1.2.x。本 Playbook 的命令和 IAM 模型若超过 12 个月未复核，请先在测试环境验证 SG / IAM Group 的最新行为。
