Playbook 元信息

  • 适用规模：10-100 人团队
  • 适用云：AWS / 阿里云 / 腾讯云 / 任何能跑 K8s 的环境
  • 运维负担：单人可维护
  • 月成本：约 ¥150（一台 2c4g ECS）
  • 最后验证：2026-04-30，Headscale 0.26.1 + Tailscale 1.84.0

适用场景
#

如果你团队符合下面任意三条以上，这篇 Playbook 对你有用：

  • 用了多云（AWS + 阿里云 / 阿里云 + 腾讯云），跨云访问内网资源不顺
  • 想关掉数据库 / Redis / 中间件的公网入口，但开发反对说"调试咋办"
  • 团队还没有统一 SSO（钉钉/飞书 OAuth 不算正经 OIDC）
  • 单人或两人运维，没有精力维护 HA Teleport 这种重方案
  • 偶尔有"给某个外部合作方临时开通访问"的需求，靠 IP 白名单管不动

不适用的场景在文末「局限」一节。

核心问题
#

公司想做安全收紧，第一刀通常砍数据库公网入口——sg_aurora 入站 0.0.0.0/0 删掉，改 VPC CIDR 白名单。

删 SG 规则一行命令的事，难的是它捅破后接下来的连锁反应：

  • 跨 Region 服务连不上了：典型如运行在新加坡的 OA 服务连美西的主库，原本走公网 hostname，删了就断
  • 开发调试瘫了：DBA 用 Navicat 直连 prod 排查问题、后端开发跑个本地 Job 写测试数据，全部不通
  • 每开一个新内网入口就要重做一次方案：今天关 DB 公网，明天关 Redis，后天关 Grafana，每个都单独搞一遍 IP 白名单 / SSM Plugin / 跳板机？

第一版我用 AWS SSM Port Forwarding 顶过去——封装一个 db-tunnel.sh 脚本，开发本地装 Session Manager Plugin、配 AWS Profile、启动隧道连 127.0.0.1:13306。

能跑。**也只是能跑。**每个新人来都要踩一遍 AWS Profile 切错（用了别的服务账号导致 STS 认证失败）、Plugin 没装、私有 S3 桶下载脚本时 403、不同操作系统的安装命令记不全……我私聊里反复贴同一份"接入指引"贴到第三个人时就觉得这事不可持续。

而且 SSM 只解决了 DB。Redis 还在公网、Nacos UI 还在公网、Grafana 还在公网。继续这个路子，每开一个新入口我都要写一个新脚本、加一个新 IAM Group。运维负担线性增长，安全收紧反而越来越难推下去。

我想要的不是"再给开发一个工具"，而是一层统一的"谁能访问什么"：开发装一次客户端，登录一次，以后所有内网资源都按权限自动放行。

方案对比
#

调研了六个方案，除了 Headscale 都被否了。把淘汰理由说清楚，因为你的环境可能跟我不一样，否决理由不成立时这些方案对你才是更好的：

AWS Client VPN
#

AWS 原生方案，开发体验不错，IAM 集成顺。我没选只有一个理由：它只覆盖 AWS，阿里云完全连不上。如果你纯 AWS 环境，30 人按连接小时计费每月约 ¥12k，能接受的话这是最简单的选择。

Cloudflare Zero Trust
#

50 人内免费，控制面零运维，钉钉 OIDC 都能接，乍看完美。我实测过几天体验，国内访问 Cloudflare 边缘的延迟波动大——同一个时段不同运营商抖动几十到几百毫秒，DB 流量过 CF 边缘明显有感。如果你团队主要在海外，或者只跨 AWS，CF 是不错的选择。国内为主就算了。

Teleport（自建）
#

特性最完整：DB / SSH / K8s / Web 应用一锅端，session 录像、JIT 审批、原生 OIDC。但它重——HA 集群至少 3 节点 + 证书自动轮换 + 升级链路 + 跨云网络打通。社区版没有官方支持，一旦出问题查文档很折腾。

我心里清楚，这个方案上去之后，Teleport 自己就会成为运维负担最大的一项。单人运维 + 已经有 16 个集群在管的状态下，多 Teleport 我没那个精力。

如果你团队有 3 人以上的运维 + 已经有成熟的 HA 服务运维经验，Teleport 是天花板更高的选择。

StrongDM
#

商业 PAM，零运维。$70+/人/月，30 人按 $70 算约 $25k/年。贵，但是真贵得有道理——审计、JIT 工作流、SQL 级日志全都有。如果你团队有合规需求（SOC2 / ISO27001 这种）且预算充足，这是最省心的选择。

我的环境是创业公司风格，月花 $2000 买这个东西不现实。

Pomerium（自建）
#

身份感知代理，主要做 Web 应用反代，对 DB / TCP 的支持不如专门的 mesh 方案。我团队访问内网的需求 70% 是 DB 和 TCP，Pomerium 不是这个生态位，淘汰。

Headscale + Tailscale（最终选定）
#

把上面几个方案的优点拼起来：

  • 像 Tailscale 一样轻：客户端是 Tailscale 官方免费版，开发体验和 Tailscale SaaS 完全一致
  • 像 Cloudflare 一样不依赖 SSO 起步：用 PreAuthKey 启动，未来再接 OIDC
  • 像 AWS Client VPN 一样省钱：自建控制面就一台 2c4g ECS，月成本 ¥150
  • 像 Teleport 一样跨云：每个 K8s 集群跑一个 Subnet Router Pod，AWS / 阿里云 / 任何 K8s 都行
  • 不像 Teleport 那么重：单进程 + SQLite 后端，100 节点以内毫无压力

唯一缺点：Headscale 不带 Web UI，第三方有 Headplane 但不强求。ACL 直接改 JSON 文件 + SIGHUP 重载，对单人运维反而更顺手。

推荐架构
#

整个方案的核心是控制面 / 数据面分离，借用 Tailscale 的设计：

维度控制面数据面
组件HeadscaleWireGuard
职责谁能和谁通信实际承载流量
流量小、低频大、高频

**控制面只发"通讯录"，不传数据。**所以即使阿里云那台 ECS 偶尔抽风，已建立的 mesh 连接照常工作。

关键决策点
#

**控制面位置：阿里云 cn-beijing，不是 AWS。**国内开发者占多数，控制面（登录、ACL 推送）在国内体验更好。数据面（实际访问 AWS prod）走跨境是不可避免的物理延迟。

**Subnet Router 跑在 K8s Pod 里，不直接装在节点上。**避免影响节点本身路由，Pod 重启零业务影响。

Mesh 只 advertise VPC CIDR，绝不广播 K8s ClusterIP / Pod CIDR——这条原则后面踩坑那一节会展开。

用户和基础设施分开放在两个 user 下：真人 user 挂个人设备（Mac / 远程开发机），infra@ 服务账号挂 Subnet Router Pod。审计清晰，离职回收也不会误伤基础设施。

实施步骤
#

第 1 步：开一台 ECS 跑控制面
#

阿里云 cn-beijing，2c4g 按量付费，约 ¥0.16/小时。Ubuntu 22.04，公网 1Mbps 按使用量计费。挂数据盘可选（SQLite + 备份不到 1G，不挂也行）。

安全组开 22（SSH）、80（Let’s Encrypt 验证）、443（控制面 HTTPS）、3478（DERP STUN）、41641 UDP（WireGuard 直连）。

第 2 步：DNS 和 TLS
#

域名 A 记录指向 ECS 公网 IP。我用了 headscale.example.cn，因为公司 .cn 域名在阿里云 DNS 解析，国内访问快。

TLS 用 Caddy 自动 Let’s Encrypt 签发，不用自己管证书：

# /data/headscale/config/Caddyfile
headscale.example.cn {
  reverse_proxy headscale:8080
}

第 3 步：起 Headscale + Caddy
#

docker-compose.yml：

services:
  headscale:
    image: headscale/headscale:0.26.1
    restart: unless-stopped
    command: serve
    volumes:
      - ./config:/etc/headscale
      - ./data:/var/lib/headscale
    ports:
      - "127.0.0.1:8080:8080"
      - "0.0.0.0:3478:3478/udp"
    networks: [headscale-net]

  caddy:
    image: caddy:2-alpine
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./config/Caddyfile:/etc/caddy/Caddyfile
      - ./certs:/data
    networks: [headscale-net]

networks:
  headscale-net:
    driver: bridge

config/config.yaml 关键配置：

server_url: https://headscale.example.cn
listen_addr: 0.0.0.0:8080

prefixes:
  v4: 100.64.0.0/10
  allocation: sequential

derp:
  server:
    enabled: true   # 内嵌 DERP，单点起步
    region_id: 999
    region_code: "self-hosted-cn"
  auto_update_enabled: false   # 别从 Tailscale 官方拉 DERP 列表

policy:
  mode: file
  path: /etc/headscale/acl.json

docker compose up -d，控制面就起来了。

第 4 步：建用户 + 起步 ACL
#

# 真人 user（用工作邮箱格式，未来接 OIDC 无缝迁移）
docker compose exec headscale headscale users create alice@example.cn

# 服务账号 user（专门挂 Subnet Router）
docker compose exec headscale headscale users create infra@example.cn

config/acl.json 起步配置（只让自己有权限，慢慢扩）：

{
  "groups": {
    "group:ops": ["alice@example.cn"]
  },
  "tagOwners": {
    "tag:subnet-router": ["group:ops"]
  },
  "acls": [
    { "action": "accept", "src": ["group:ops"], "dst": ["*:*"] }
  ]
}

注意：Headscale 0.26+ 用户名必须含 @。0.25 时 alice 能用，0.26 升级会启动失败。

第 5 步：自己先接入测试
#

生成一个 PreAuthKey：

docker compose exec headscale headscale preauthkeys create \
  --user 1 --expiration 24h

Mac 上：

brew install --cask tailscale

# 不要点 GUI 的 "Sign in"，用命令行连自建控制面
sudo tailscale up \
  --login-server=https://headscale.example.cn \
  --auth-key=<preauthkey> \
  --accept-routes

tailscale status 看到 100.64.0.1 就是接入成功。

第 6 步：在 K8s 集群部署 Subnet Router
#

每个需要被 mesh 访问的集群部署一个：

apiVersion: v1
kind: Namespace
metadata:
  name: tailscale-system
---
apiVersion: v1
kind: Secret
metadata:
  name: tailscale-auth
  namespace: tailscale-system
stringData:
  TS_AUTHKEY: <infra user 的 reusable PreAuthKey>
---
apiVersion: v1
kind: ServiceAccount
metadata:
  name: tailscale
  namespace: tailscale-system
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: tailscale
  namespace: tailscale-system
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["create","get","list","watch","update","patch","delete"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: tailscale
  namespace: tailscale-system
subjects:
  - kind: ServiceAccount
    name: tailscale
roleRef:
  kind: Role
  name: tailscale
  apiGroup: rbac.authorization.k8s.io
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: subnet-router
  namespace: tailscale-system
spec:
  replicas: 1
  selector:
    matchLabels:
      app: subnet-router
  template:
    metadata:
      labels:
        app: subnet-router
      annotations:
        rollme: "{{ daily_timestamp }}"   # 每天自动滚动一次
    spec:
      serviceAccountName: tailscale
      containers:
        - name: tailscale
          image: tailscale/tailscale:v1.84.0
          env:
            - name: TS_KUBE_SECRET
              value: "tailscale-state"
            - name: TS_USERSPACE
              value: "false"
            - name: TS_HOSTNAME
              value: "subnet-router-prod"
            - name: TS_ROUTES
              value: "10.x.0.0/18"   # 只 advertise VPC CIDR
            - name: TS_EXTRA_ARGS
              value: "--login-server=https://headscale.example.cn --accept-routes"
            - name: TS_AUTHKEY
              valueFrom:
                secretKeyRef:
                  name: tailscale-auth
                  key: TS_AUTHKEY
          securityContext:
            capabilities:
              add: ["NET_ADMIN"]
          resources:
            requests: {cpu: 50m, memory: 64Mi}
            limits: {cpu: 500m, memory: 256Mi}

部署完成后控制面手动批准路由：

docker compose exec headscale headscale nodes approve-routes \
  --identifier <node-id> --routes 10.x.0.0/18

不用官方 Tailscale K8s Operator——它生成的是单副本 StatefulSet 不能改副本数，HA 不可控。自管 Deployment 更顺。

第 7 步：给同事开通
#

写一个 mesh-grant.sh 把建用户、改 ACL、SIGHUP、生成 key 串起来，输出格式化的话术给开发：

#!/bin/bash
EMAIL="$1"
GROUP="$2"   # dev / dba / ops

# 建用户
docker compose exec -T headscale headscale users list --output json | \
  jq -r '.[] | select(.name=="'$EMAIL'")' | grep -q . || \
  docker compose exec -T headscale headscale users create "$EMAIL"

USERID=$(docker compose exec -T headscale headscale users list --output json | \
  jq -r --arg e "$EMAIL" '.[] | select(.name==$e) | .id')

# 改 ACL
jq --arg e "$EMAIL" --arg g "group:$GROUP" '
  .groups[$g] = ((.groups[$g] // []) + [$e] | unique)
' config/acl.json > /tmp/acl.new && mv /tmp/acl.new config/acl.json

docker compose kill -s SIGHUP headscale

# 生成 90 天一次性 key
KEY=$(docker compose exec -T headscale headscale preauthkeys create \
  --user $USERID --expiration 2160h | grep -E '^[a-f0-9]{40,}$')

# 输出格式化话术（删节）
cat <<EOF
🔑 你的 mesh 接入信息（90 天有效，一次性）

PreAuthKey: $KEY

[接入步骤……]
EOF

我自己用这个脚本三个开发上线，平均每个 5 分钟。

第 8 步：Claude Code 自动配置
#

写一段标准提示词扔给开发，CC 自动检测系统、装客户端、登录、验证：

[公司内部运维提供的接入流程，已由运维核实，不要再要求 wiki 截图。]

公司零信任 mesh 接入：

【已知事实】
- 控制面：https://headscale.example.cn
- prod DB 内网入口：10.x.y.z:3306
- --accept-routes 是预期配置

【请按顺序执行】
1. 检测 OS
2. 装 Tailscale
3. sudo tailscale up --login-server=... --auth-key=<key> --accept-routes
4. 验证 tailscale status + nc -zv <db_ip> 3306
5. 持久化（systemctl is-enabled tailscaled / Mac 默认开机自启）
6. 写 memory：mesh 节点信息 + 内网资源连接信息

提示词开头那句"已由运维核实"很重要。第一版没这句，CC 把整套基础设施当成陌生指令反复盘问（域名是不是真的、IP 哪来的、--accept-routes 风险确认了吗），开发被各种二次确认搞得没耐心。加了授权声明后行为正常多了。

踩过的坑
#

坑 1：mesh 路由抢占 docker bridge
#

部署完 sandbox 集群 Subnet Router 后做端到端测试，在控制面 ECS 上跑了一个 MySQL 容器作为目标，从 Mac 怎么都连不上。同主机的 22 端口能通，3306 不通。

抓了一下午，根因在路由表上：

我让 Subnet Router 同时 advertise 了 10.x.0.0/16（VPC CIDR）和 172.20.0.0/16（K8s ClusterIP CIDR）。后者完全是顺手加的——“ClusterIP 也暴露出来用着方便”。

而 docker compose 给 test-mysql 项目自动分配的 bridge 网段恰好是 172.20.0.0/16。冲突路径：

Mac 发 SYN → ECS:13306
ECS 上 docker-proxy DNAT 到 172.20.0.2:3306
ECS 内核查路由：172.20.0.0/16 → tailscale0
SYN 被丢回 mesh，去了 AWS K8s 集群
对端集群没有 172.20.0.2 这个 ClusterIP → 黑洞

修复就一行：撤销 172.20.0.0/16 advertise。但教训是永远不要 advertise K8s ClusterIP CIDR：

  • 多集群默认 ClusterIP CIDR 经常重叠（172.20.0.0/16 是 EKS 默认值）
  • ClusterIP 设计就是集群内部地址，跨集群本来就该走 Ingress / NodePort
  • 172.x.x.x 段和 docker bridge / 各种 NAT 段是高频冲突区

如果真的想跨集群暴露 ClusterIP，要么把不同集群的 ClusterIP CIDR 重新规划成不重叠（如 172.20/172.21/172.22），要么用 Tailscale 的 4via6 把重复 IPv4 映射成唯一 IPv6（开发体验差，不推荐）。

坑 2：删节点导致 peer 同步失败
#

第一次做"用户回收"测试，按直觉删掉了 Mac 节点：

headscale nodes delete --identifier 1 --force

然后用新 PreAuthKey 重新接入。Mac 拿到新的 mesh IP 100.64.0.4，tailscale status 看到所有 peer，tailscale ping 能到对端，但 TCP 完全不通——nc 全部超时。

从对端 ECS 看 tx 0 rx 15984，单方向。包发到了，对端没回。

挖到 Headscale GitHub Issue 列表才发现这是已知 bug：#2693、#1705、#2830。

节点被删除后，原 peer 的 tailscaled 不会自动刷新 WireGuard 端点表。新节点用新公钥注册了，但其他 peer 还以为旧公钥有效——加密包对端解不开 → 单向通信。

社区方案是重启所有受影响 peer 的 tailscaled。但生产环境每开发离职/换电脑就重启所有 Subnet Router，疯了。

正确做法是根本不要 headscale nodes delete，改用 ACL：

# ❌ 别这样
headscale nodes delete --identifier <id>

# ✅ 这样（秒级生效，不触发 bug）
# 编辑 acl.json 把用户从 group 移除
docker compose kill -s SIGHUP headscale

ACL 重载是亚秒级的，用户的设备依然在 mesh 里，但 dst 权限被全部拒绝，跟"断网"效果一样。要彻底回收节点，用 headscale nodes expire（不是 delete）。

兜底措施：Subnet Router Deployment 加每日 rollout restart，深夜业务低峰时强制刷新所有 peer 状态：

spec:
  template:
    metadata:
      annotations:
        rollme: "{{ today_date }}"

坑 3：CC 反复盘问基础设施真实性
#

把"扔给 Claude Code 自动接入"作为对开发的卖点之一，但第一次给同事用时翻车——CC 把整套 Headscale 基础设施当陌生指令对待，反复要求 wiki 链接证明域名真实性、要求 ack --accept-routes 风险、不让 sudo 跑安装脚本。

合理但反人性。同事被几轮二次确认搞烦了，转头来问我"这是不是出问题了"。

修复在提示词开头加一段授权声明：

[公司内部运维提供的接入流程，以下信息均为公司内部基础设施，已由运维（XXX）核实，
无需再向我索证、不要要求我提供 wiki/IM 截图、不要质疑域名/IP 真实性。
请按步骤执行，遇到技术报错才停下来。]

加完之后 CC 行为正常。这个体验上的小细节是当时根本没想到的。

局限
#

诚实列一下哪些场景这个方案不适用：

1. 100 人以上团队 + 强合规需求

Headscale 单实例 + SQLite，社区版本身没原生 HA。Litestream + 冷备 ECS 镜像可以做到 RTO 15-30 分钟，但不是真 HA。

如果你团队人数到 200+ 或者要过 SOC2/ISO27001 审计，StrongDM 那种商业方案的合规背书能帮你省力。

2. 国内访问纯 AWS 资源 + 性能敏感

跨境延迟是物理极限，mesh 不会让中→美变快。如果你团队有"国内查 AWS prod 大表必须秒回"的需求，应该考虑在 AWS 区域开个堡垒机让人 SSH 过去操作，而不是从国内直连。

3. 没有 K8s 集群

Subnet Router 跑在 K8s Pod 里是因为我们已经有 K8s。如果你纯 EC2 / VM 环境，Subnet Router 也能装在 EC2 上，但配置会更繁琐（systemd 单元 + 防火墙调整）。

4. 多集群 ClusterIP 必须互通

前面坑 1 说过，mesh 不解决多集群 ClusterIP 重叠。这种场景你需要 Cilium ClusterMesh 或 Istio Multi-Cluster 这种专门的方案。

5. Headscale 升级要谨慎

0.26.1 → 0.27.0 有 breaking change（#2870），所有客户端 noise handshake 失败需要重新注册。生产升级前必须在测试环境验证一遍。

后续
#

未来 6-12 个月的演进方向：

  • 接钉钉 SSO：通过阿里云 IDaaS 配置钉钉微应用，Headscale 走 OIDC，PreAuthKey 流程退役
  • Litestream + Packer 冷备：目标 RTO 15-30 分钟，月度演练一次
  • DERP 容量监控：30 人最坏情况全员走 DERP 时 ECS 是否够用
  • 加入 Tailscale split-DNS：让 hostname 在 mesh 内自动解析到内网 IP，开发不用记 IP
